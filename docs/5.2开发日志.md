## RMSNorm 设计与实现

### 算法回顾

```
RMS(x) = sqrt((1/d) * Σ x_i² + ε)
y_i = (x_i / RMS(x)) * γ_i
```

### 并行策略

1 block × 1 warp × 32 thread,每 lane 负责 28 个连续元素(d=896,896/32=28 整除)。

### 数据流分四阶段

- **Phase A**:每 lane 累计局部平方和(28 个元素)
- **Phase B**:5 步 xor 树形归约(`VSHFL_XOR` mask = 1, 2, 4, 8, 16),全 lane 拿到 total_sumsq
- **Phase C**:rrms = 1 / sqrt(total_sumsq * inv_d + eps)
- **Phase D**:每 lane 第二轮遍历自己的 28 个元素,做 `out[i] = in[i] * rrms * weight[i]`

### 几个设计决策

1. **inv_d 由 host 预算**:`d=896` 是常量,host 算 `1.0f/d` 塞进 args struct,kernel 直接用,避免每 warp 算 32 次相同除法
2. **rrms 提到循环外**:Phase C 算一次,Phase D 全用乘法(28 次 fdiv → 1 次 fdiv,显著提速)
3. **fully converged 保证**:Phase A 的循环次数是编译时常量 28,所有 lane 同步进同步出,Phase B shuffle 安全
4. **连续分块 vs strided**:在纯解释器上没有 coalescing,选连续分块(索引算式更简单)

### args 结构体

```c
struct rmsnorm_args {
    uint32_t in_off;
    uint32_t out_off;
    uint32_t weight_off;
    uint32_t d;
    float    inv_d;
    float    eps;
};
```

## libgpgpu slab 分配器重构

### 触发原因

写 RMSNorm 的 host 测试时,`gpuMalloc(3584)` 返回失败(`slab_alloc failed`),导致 `weight_off = (uint64_t)-1`,后续 segfault。

### Bug 链分析(共三个独立 bug)

**Bug A:size class 路由错误**

```c
if (size < 4096) {
    *marker_out = PAGE_ORDER_SLAB_2048;
    return &ctx->slab_2048;
}
```

`route_slab` 把 `(2048, 4096)` 区间的 size 路由到 `slab_2048` —— 但 slab_2048 的 slot size 是 2048。3584 字节请求被分配 2048 字节的 slot,`gpuMemcpy` 拷贝时**写穿到下一个 slot**,污染了 slab 内部 metadata。

**Bug B:slab 容量不可扩展**

```c
int slab_cache_init(...) {
    uint64_t offset = buddy_alloc(buddy, PAGE_SIZE);  // 只申请一个 page
    ...
}
```

每个 size class 只有一个 backing page。`slab_2048` 一个 page = 2 个 slot,**第三次 alloc 必然 free_list 为空,直接返回 -1**。

**Bug C:slab_alloc 没有从 buddy 扩容的 fallback**

```c
uint64_t slab_alloc(struct slab_cache *cache) {
    if (!cache->free_list) return -1;  // 直接放弃
    ...
}
```

Bug A 和 B/C 是分开存在的——3584 字节路由错误 + slab 不能扩容,叠加触发了 RMSNorm 这次失败。

### 设计:slab V2

新增数据结构 `slab_page_meta`:

```c
struct slab_page_meta {
    uint64_t page_offset;
    uint16_t free_count;
    uint16_t total_slots;
    struct slab_page_meta *next;
};

struct slab_cache {
    size_t object_size;
    uint8_t marker;
    struct buddy_allocator *buddy;
    struct slab_node *free_list;        // 全局,横跨所有 page
    struct slab_page_meta *pages;       // 每个 page 一个 meta,链表组织
};
```

**关键洞察**:free_list 和 pages 是**两个独立视角**——free_list 服务 alloc/free 的 O(1) 操作,pages 服务回收决策(`free_count == total_slots` 触发整页归还)。

**新增三个 helper**:
- `slab_grow(cache)`:从 buddy 拿一个 page,malloc meta,切片头插到 free_list
- `find_meta_for_offset`:`offset & ~(PAGE_SIZE - 1)` 找到 page 边界,扫 pages 链表匹配
- `slab_shrink`(在 `slab_free` 内联):整页全空时,扫 free_list 摘掉所有该页 slot,从 pages 摘 meta,buddy_free 归还,free meta

**关键约束**(踩了一个坑后修):**保留至少 1 个 page 不归还**,避免 alloc → free → alloc 抖动:

```c
if (meta->free_count == meta->total_slots && has_other_page(cache)) {
    // shrink
}
```

否则 `test_slab` 这种 "alloc 一次 + free 一次" 的最小测试会触发 shrink → free_list 变 NULL → assert 失败。

### 配套修改

1. `route_slab`:`size < 4096` → `size <= 2048`,> 2048 直接走 buddy
2. `gpuInit`:slab_cache_init 加 marker 参数,删掉 6 行手动设 page_order
3. `slab_cache_init`:不再自己 buddy_alloc,改成调 slab_grow 申请第一个 page

### 验证结果

- `test_buddy` ✅
- `test_slab` ✅(修了 has_other 判断后)
- `test_fmadd` 一开始 FAIL,但跟 slab 无关——见下

## 潜伏多日的 QEMU bug:KERNEL_ARGS MMIO 写无人处理

### 现象

slab 重构跑通后,试 `test_fmadd` 验证回归——**FAIL**,32 个 C 元素全是 host 上传时的初始值 1.0,说明 kernel 的 `fsw` 没生效。dmesg 干净,反汇编正确,`fmadd.s` 编码 `0x1820f243` 对应 opcode 0x43,无误。

### 排查路径

**Step 1**:host 直接读 `vram[c_off]`,仍是 1.0 → kernel 没写,不是 D2H 拷贝问题。

**Step 2**:kernel 末尾加 debug `sw`,把 a0/a3/t4 写到已知位置,host 读出来:

```
[DEBUG] a3 (expect c_off=0x2e80) = 0x40000000
[DEBUG] t4 (expect c_off+0)    = 0x40000000
[DEBUG] a0 (expect args_off)   = 0x40000000
```

`0x40000000` 看似神秘,但识别出**这是 IEEE 754 浮点 2.0f 的位模式**——也就是 `ha[i] = 2.0f` H2D 上传后写在 vram[a_off] 的原始数据。说明 kernel 的 debug `sw a3, 0(a1)` 根本没改 vram,反推 a1 错了 → a0 错了。

**Step 3**:查 `gpgpu_ctrl_write`,发现 `case GPGPU_REG_KERNEL_ARGS_LO` 和 `_HI` **完全不存在**,switch 落到 `default: break;`,驱动写过来的 args 地址被静默丢弃。`s->kernel.kernel_args` 永远是 `gpgpu_reset` 时 memset 的 0。

`gpgpu_core_exec_kernel` 里 `warp.lanes[j].gpr[10] = (uint32_t)s->kernel.kernel_args` 把 0 给了 a0。

### 为什么之前 fmadd 测试通过过?

bug **一直存在**,被一个巧合掩盖。

旧 slab(单 page 容量)+ 旧切片顺序下,第一次 alloc 拿到的 offset 恰好是 vram[0]。args 24 字节路由到 slab_32,旧切片是顺序排列、第一个被取走的是 offset=0 的 slot——**args_off = 0**。

QEMU 这边 args=0(因为 case 漏写,默认值)。a0 = 0。kernel `lw a1, 0(a0)` 读 vram[0],vram[0] 就是 args 本身。**两个错相互抵消**。

slab V2 改成头插,第一次 alloc 拿到的是 offset=127*32=0xfe0(slab_32 page 的最后一个 slot)。args 数据放在 vram[0xfe0],kernel 仍用 a0=0 读 vram[0],拿到的是 slab_32 那块 page 上的 free_list 节点指针——完全是垃圾值。

**slab 重构没引入 bug,它揭露了潜伏的 bug。**

### 修复

`gpgpu_ctrl_write` 加两个 case:

```c
case GPGPU_REG_KERNEL_ARGS_LO:
    s->kernel.kernel_args = (s->kernel.kernel_args & 0xFFFFFFFF00000000ULL) | val;
    break;
case GPGPU_REG_KERNEL_ARGS_HI:
    s->kernel.kernel_args = (s->kernel.kernel_args & 0xFFFFFFFF) | ((uint64_t)val << 32);
    break;
```

修复后 `test_fmadd` PASS,debug 输出 `a3 = 0x2e80 = c_off`,完全对上。

## 经验沉淀

**1. 巧合掩盖的 bug**
当某个值"恰好为 0"或"恰好对齐",代码看起来正确,只是输入空间的某个角落命中了正确路径。两个独立错误互相抵消能跑通,这种 bug 在**地址映射、序列化协议、硬件 MMIO** 这类有"约定值"的层面尤其常见。

**2. 浮点位模式直觉**
快速识别浮点常见值的 IEEE 754 表示能极大加速 debug:

| float | hex |
|---|---|
| 1.0 | 0x3F800000 |
| 2.0 | 0x40000000 |
| 3.0 | 0x40400000 |
| 32.0 | 0x42000000 |

调试浮点 kernel 看到 hex 值时,先扫一眼是不是这些常见值的位模式。

**3. 沿数据流逆推,每步用 dump 验证**

```
vram[c_off] 错
 → fsw 写错
 → t4 错(kernel 内 dump 验证)
 → a3 错
 → a0 错(kernel 内 dump 验证)
 → init_warp 时 a0 = s->kernel.kernel_args = 0
 → MMIO 写没生效
 → switch 漏 case
```

每步 dump 验证,不靠猜。

**4. MMIO read/write 必须对称**
`gpgpu_ctrl_read` 也漏了 KERNEL_ADDR/ARGS 的 case。如果驱动哪天想 read-back 验证,会读到 0,误判没写过。Read/write 不对称是 MMIO 设备模型常见的潜伏 bug。

**5. 工程化的演进路径**
slab V1 → V2 的重构很值得——V1 设计上就不可扩展(每 cache 一个 page),撞到 RMSNorm 这种新 size 立刻露馅。但 V2 也没做完美:**没有把 page 还给 buddy 之外的"page 缓存复用"机制**(刚还的 page 立刻又能被同 cache 拿回来),没有 partial/full 链表分桶(shrink 时 O(N_total) 扫整个 free_list)。这些是 V3 的事。

经验:**先 working,再 clean,再 fast**。今天到 working 就够。
