## 本次会话主要工作

### 1. Softmax 算子设计与实现

**算法**:
```
m = max(x)
y_i = exp(x_i - m) / Σ exp(x_j - m)
```

**并行策略**:沿用 RMSNorm 的 1 warp × 32 lane × 28 elem/lane(d=896,896/32=28 整除)。

**三阶段数据流**:
- **Phase A**:每 lane 求 local_max,5 步 xor warp shuffle 归约 max → global_max
- **Phase B**:每 lane 算 `exp(x_i - max)` 并累计 local_sum,warp shuffle 归约 sum → global_sum
- **Phase C**:每 lane 重算 `exp(x_i - max)` 并 `out[i] = exp(...) / sum`

**关键设计决策**:
1. **3-pass 标准版**(非 online softmax):d=896 不大,3-pass 足够,online softmax 数值更新公式易写错,留待 attention 阶段考虑
2. **scaled dot product 的 scale 留到 attention 决策时再定**:Softmax 独立算子不内置 scale
3. **没有 inv_d / eps**:减 max 后 exp 范围 (0, 1] 数值天然稳定,不会溢出/下溢到错误结果
4. **local_max 用 -INF 初始化**:支持负值输入(`fmv.s.x ft0, x_with_0xFF800000`),不能用 0 初始化(后续 attention logits 会出现负值)
5. **fmax.s 顺手实现**(funct7=0x14, funct3=0/1):RV32F 标准指令,RMSNorm 不用但 Softmax 必须有

### 2. 自定义指令 vexp.s 设计与实现

**编码**:funct7=0x7c, rs2=0, funct3=0, opcode=0x53(OP-FP)

**实现**:`exec_fp` 内调用 host `expf()`:
```c
else if (funct7 == 0x7c) {
    uint32_t a = lane->fpr[rs1];
    float fa;
    memcpy(&fa, &a, 4);
    float fr = expf(fa);
    uint32_t r;
    memcpy(&r, &fr, 4);
    lane->fpr[rd] = r;
}
```

**汇编宏(用 `.word` 拼,绕开 GAS 对 `.insn r` 的不一致检查)**:
```asm
.macro VEXP_S rd, rs1
    .word (0x7c << 25) | (0 << 20) | (\rs1 << 15) | (0 << 12) | (\rd << 7) | 0x53
.endm
# 调用:VEXP_S 2, 1   # vexp.s f2, f1 — 传整数寄存器编号
```

**编码选择经验**:
- 第一次选 funct7=0x60 → **撞上 RV32F 标准的 fcvt.w.s**(spec 占用,但 exec_fp 已实现成自定义 op,silently shadow 了标准指令)
- 改用 funct7=0x7c(spec 中 0x78 的下一个,无标准指令占用)
- **教训**:加自定义指令前必须核对 RV32F spec funct7 占用表

### 3. exec_fp 加 default case + funct7 占用表注释

**default case**:

else {

### 4. SiLU 算子设计与实现

**算法**:`silu(x) = x / (1 + exp(-x))`

选直接除形式而非 `x * sigmoid(x)` 形式——少一次 fmul,数值等价。

**并行结构**:**纯 element-wise,无 reduce,无跨 lane**——element-wise 算子模板的最简实例。

```asm
loop:
    flw     ft1, 0(t2)        # x
    fsgnjn.s ft2, ft1, ft1    # -x  (= fneg.s 伪指令展开)
    VEXP_S  2, 2              # exp(-x)
    fadd.s  ft2, ft4, ft2     # 1 + exp(-x)  (ft4 是循环外预装的 1.0)
    fdiv.s  ft3, ft4, ft2     # sigmoid(x)
    fmul.s  ft0, ft1, ft3     # x * sigmoid(x)
    fsw     ft0, 0(t3)
    ...
```

**关键设计点**:
1. **fneg via fsgnjn.s**:取负只用 fsgnjn 一条指令翻转 sign bit,无需 fzero,无需常量 -1.0
2. **常量 1.0 提到循环外**(`li t4, 0x3F800000; fmv.s.x ft4, t4`):跟 RMSNorm rrms 提到外面是同一个优化思路在迁移
3. **x 必须独立保留在 ft1**:fmul 时还要用,寄存器分配避免被中间值覆盖——这是 element-wise 算子的"x 不要被覆盖"陷阱,attention 里会反复出现

**测试结果**:
- max_err = **9.54e-7**(略大于 RMSNorm/Softmax 的 1e-9 量级,源于 `1 + exp(-x)` 在 x 大时接近 1.0 时 fdiv 的 ULP 损失,IEEE 单精度物理下限,非 bug)
- 数值 spot check:`silu(-8.96) ≈ -0.00115` ✓,`silu(8.94) ≈ 8.939` ✓

## 关键经验沉淀(累积到 V12 之上)

### 关于"通过 ≠ 正确"的测试设计陷阱

Softmax 第一次"PASS"是**假阳性**。原始测试输入是 `h_in[i] = (i % 7) * 0.1f`,周期 7,而 lane stride = 28 = 4×7,**导致每个 lane 的 28 个元素分布完全相同,所有 lane 的 local_max / local_sum 全部相等**。这种输入下:
- warp shuffle 归约 max:每 lane 已经是相同值,即使 shuffle 完全错,fmax 结果也是同一个值
- warp shuffle 归约 sum:每步加邻居,邻居和自己一样,5 步翻倍 ×32 = 真实 global_sum,**碰巧凑齐正确答案**

换成 `(i - 448) / 50.0f` 后(每 lane 范围不重叠、max 在最后一个 lane),reduce 必须真正跨 lane 才能算对 → 这才是真测试。

**沉淀的原则**:
- **测试输入要打破对称性**:周期性输入跨 lane 边界要刻意检验"是否制造了 lane-stride 的整数倍周期"
- **小心"sum check 全为 1.0"这类弱断言**:softmax sum=1 是必要不充分条件
- **新算子用至少两套差异显著的输入测**(我们用了 `(i-448)/50` 验,以后 attention 也照办)

### 关于自定义指令的 funct7 选择

**0x60 撞 fcvt.w.s 事件**揭示了一个系统性问题:

> "exec_fp 里这个 funct7 没实现"≠"RV32F spec 里这个 funct7 空闲"

之前没人写 fcvt.w.s 测试,所以自定义 op 把它 shadow 了也没人发现。**未来加自定义指令必须查 RV32F spec 的 funct7 占用表**(已写到 exec_fp 头注释)。

### 关于 RV32 的 GAS 边界

- **整数寄存器只到 t6**(t0-t2 + t3-t6 共 7 个),**没有 t7..t9**
  - 错误信息是 `illegal operands`,容易误导成"操作数类型错"——根因是寄存器名不存在
- **`fmv.w.x` vs `fmv.s.x`**:你这版 GAS 认 `fmv.s.x`(老名),不认 `fmv.w.x`(spec 新名)。**项目内统一用 `fmv.s.x`**
- **`.insn r` 跨版本不一致**:之前用过浮点寄存器名报错,现在加 funct7=0x7c 时 GAS 又不接受高位 funct7。**长期方案:全部用 `.word` 拼**,绕开 GAS 检查
- **`fneg.s` 是 `fsgnjn.s rd, rs, rs` 的伪指令**:认不认看版本,fallback 写显式 fsgnjn 总对

### 关于工程节奏

- **逐 phase 验证不可省**:V11 教训重申。Softmax 我们没分 phase 验证(直接整个 kernel 写完),如果第一次假 pass 不是因为周期对称巧合而是因为真 bug,debug 会非常痛苦。**Attention 必须严格 phase-by-phase**
- **测试模板有惯性**:从 softmax 复制到 silu,"sum expect 1.0"这种**算法-specific 的注释要校准**,否则误导未来读者
- **死代码要么删要么注释**:silu kernel 的 `lw a3` 读了 D 但没用——保留时加注释说明意图("currently unused; loop trip hardcoded to 28")

### 短期:Attention(milestone 3.x 启动)

**核心算法**:
```
attention(Q, K, V) = softmax(Q @ K^T / sqrt(d_k)) @ V
```

Qwen2-0.5B 参数:
- num_query_heads = 14
- num_kv_heads = 2(GQA)
- head_dim = 64
- hidden_size = 896 (= 14 × 64)

**5 个待决策的架构问题**(下次会话开始处讨论):

1. **Fused 还是拆分**?(初始倾向:拆 3 个 kernel,复用现有 softmax)

3. **多头怎么处理**?(初始倾向:第一版只跑 1 head)
4. **KV cache 现在做不做**?(初始倾向:不做,先验证算法)
5. **scale (1/sqrt(d_k)) 在哪算**?(初始倾向:host 预算塞 args,沿用 RMSNorm inv_d 模式)

下次会话第一件事:**对齐这 5 个决策**,然后才开始写代码。
