# GPGPU 驱动开发日志 —— 里程碑 2.3：SIMT 基础设施

## 概览

为虚拟 GPGPU 实现真正的 SIMT 执行模型，包括 **warp divergence + reconvergence** 和 **warp shuffle**，让设备从 SIMD 升级为可支持复杂控制流和高效归约的 SIMT 架构。

完成后，单个 warp 内的 32 个 lane 可以走不同分支，并能用 5 步 xor 树形归约完成跨 lane 的标量聚合 —— 为 RMSNorm、Softmax、Attention 等需要归约的算子铺好基础设施。

---

## 背景与动机

之前的解释器 `exec_warp` 每个 cycle 取一条指令，广播给所有 active lane 执行。这等价于 SIMD —— 所有 lane 必须在同一 PC、走同一控制流。一旦遇到 if/else 让不同 lane PC 分裂，取指就会用错 lane 的 PC，执行流彻底乱套。

同时，归约类算子（RMSNorm 算 RMS、Softmax 算 max/sum）天然需要跨线程通信。原始方案有三条退路：

1. 所有 lane 重复算同一个归约（2× 冗余）
2. 把归约扔回 host CPU（破坏 device 完整性）
3. 多个 kernel 串联 + 显式同步（host 逻辑复杂）

最终选择**直接补全 SIMT 基础设施**，这是个能复利的投资 —— Softmax、Attention 全都受益。

## 阶段一：Warp Divergence + Reconvergence

### 设计选择

真实 GPU 用 **immediate post-dominator (IPDOM) reconvergence stack** 处理 divergence：编译器在分支处压栈"汇合点 PC"，硬件按栈顺序串行执行各 path。

但这要求工具链配合（编译器分析控制流并生成栈管理指令）。我们的 ISA 是标准 RV32，工具链不会做这个分析。

退而求其次，采用 **min-PC scheduling**（早期 GPU 也用过）：

- 每个 cycle 找出所有 active lane 中**最小的 PC**
- 当前 cycle 只让 PC 等于 min_pc 的 lane 执行
- PC 较大的 lane 自然在下一 cycle min_pc 推进上来时被选中

**关键性质**：所有 lane PC 自然汇合时（都相等），current_mask 自动恢复成 active_mask，无需显式管理栈。PC 自己就是栈。

### 已知 trade-off

min-PC 不看控制流结构，只看 PC 数值。

- **失败模式**：data-dependent 的 while 循环里，如果某个 lane 一直留在循环体内（PC 较小），其他已退出循环的 lane（PC 较大，在 exit_point 等候）会被永久饿死
- **不影响当前用例**：所有目标算子（RMSNorm、Softmax、GEMM）的循环次数都是 data-independent，所有 lane 同步进入/退出循环，只在 if/else 触发 divergence —— 这是 min-PC 的强项
- **未来路径**：等需要 data-dependent 控制流时，升级到 IPDOM 栈即可，调度入口（`exec_warp`）不变

### 实现要点

`exec_warp` 主循环改造：

```c
while (warp->active_mask && cycles < max_cycles) {
    // 1. 一次遍历同时算 min_pc 和 current_mask
    uint32_t min_pc = 0xFFFFFFFF;
    uint32_t current_mask = 0;
    uint32_t temp = warp->active_mask;
    while (temp) {
        int lane = __builtin_ctz(temp);
        uint32_t pc = warp->lanes[lane].pc;
        if (pc < min_pc) {
            min_pc = pc;
            current_mask = 1u << lane;
        } else if (pc == min_pc) {
            current_mask |= 1u << lane;
        }
        temp &= temp - 1;
    }
    
    // 2. 用 min_pc 取指
    uint32_t insn = *(uint32_t *)(s->vram_ptr + min_pc);
    
    // 3. 只对 current_mask 里的 lane 执行
    for (int i = 0; i < GPGPU_WARP_SIZE; i++) {
        if (!(current_mask & (1u << i))) continue;
        // ...原有的 dispatch 逻辑...
    }
}
```

关键细节：

- ebreak 必须清 `warp->active_mask`（不是 current_mask）—— ebreak 让 lane 整个退出，不只是这一 cycle 不参与
- branch / jal / jalr 内部已处理 PC 更新，外层 `taken_branch = 1` 阻止再 += 4

### 验证

测试 kernel `divergence.S`：奇偶 lane 分别写不同 magic number 到不同 VRAM 数组：

```asm
csrrs t0, mhartid, x0
andi  t0, t0, 0x1F
andi  t4, t0, 1
bnez  t4, odd_case      # ← 触发 divergence

# 偶数 path: 写 0xAAAA 到 out_a[lane_id]
...
j after

odd_case:
# 奇数 path: 写 0xBBBB 到 out_b[lane_id]
...

after:
ebreak
```

执行轨迹（手动推演 + 实测验证）：

| 阶段 | min_pc | current_mask 含义 | 行为 |
|------|--------|-------------------|------|
| 公共段 | 同步推进 | 全 32 lane | 正常执行 |
| 分支后偶数 path | 0x1C → 0x2C | 16 个偶数 lane | 写 0xAAAA |
| `j after` | 0x2C | 偶数 lane | PC 跳到 0x40 |
| 奇数 path | 0x30 → 0x3C | 16 个奇数 lane | 写 0xBBBB |
| 汇合 ebreak | 0x40 | 全 32 lane | 一起退出 |

Host 验证逻辑：**双向检查**（"该写的写了" + "不该写的没写"），所有 32 个位置都正确 ✅

## 阶段二：Warp Shuffle

### 指令设计

`vshfl.xor rd, rs1, mask`：lane i 的 rd 寄存器 ← lane (i ^ mask) 的 rs1 寄存器值。

xor 模式天然适合树形归约：5 步（mask = 1, 2, 4, 8, 16）后，warp 内 32 个值的归约结果出现在所有 lane 的同一个寄存器里 —— **不需要显式广播**。

### ISA 编码

| 字段 | 位 | 值 | 说明 |
|------|----|----|----|
| opcode | 6:0 | 0x0B | RV32 custom-0 编码空间 |
| rd | 11:7 | fpr 索引 | 结果寄存器 |
| funct3 | 14:12 | 0 | 留作扩展（shfl.up/down 等） |
| rs1 | 19:15 | fpr 索引 | 源寄存器 |
| rs2 | 24:20 | mask (5-bit) | 直接当立即数解释 |
| funct7 | 31:25 | 0 | 留作扩展 |

操作浮点寄存器（fpr），因为目标算子归约的都是 float。

### 解释器实现

```c
static inline void exec_vshfl(GPGPUWarp *warp, uint32_t current_mask, uint32_t insn) {
    uint32_t rd  = (insn >> 7)  & 0x1F;
    uint32_t rs1 = (insn >> 15) & 0x1F;
    uint32_t laneMask = (insn >> 20) & 0x1F;
    
    // 第 1 步：snapshot 整个 warp 的 fpr[rs1]
    uint32_t snapshot[GPGPU_WARP_SIZE];
    for (int i = 0; i < GPGPU_WARP_SIZE; i++) {
        snapshot[i] = warp->lanes[i].fpr[rs1];
    }
    
    // 第 2 步：对 current_mask 里的 lane 写回
    for (int i = 0; i < GPGPU_WARP_SIZE; i++) {
        if (!(current_mask & (1u << i))) continue;
        uint32_t src_lane = i ^ laneMask;
        warp->lanes[i].fpr[rd] = snapshot[src_lane];
    }
}
```

**snapshot 必须读全 32 lane 而不是只读 active**：lane i 的伙伴 (i^mask) 可能在 divergence 的另一条 path 上，但它的 fpr 是有意义的旧值。

**写回必须只对 current_mask**：保证 divergence 另一条 path 上的 lane 不被污染。

### 调用位置：曾经踩的坑

第一版把 `exec_vshfl` 直接放进 `exec_warp` 的 per-lane for 循环里：

```c
for (int i = 0; i < GPGPU_WARP_SIZE; i++) {
    if (!(current_mask & (1u << i))) continue;
    switch (opcode) {
        // ...
        case 0x0B:
            exec_vshfl(warp, current_mask, insn);  // ← 被调用 32 次
            break;
    }
}
```

**Bug**：`exec_vshfl` 一次调用就处理整个 warp，放在 for 循环里相当于执行了 32 次。

- 如果 rs1 ≠ rd：每次结果都一样，**碰巧正确**
- 如果 rs1 == rd（**归约的标准写法**）：第 1 次正确，第 2 次 snapshot 抓到的是已被覆盖的新值，再做一次 shuffle —— 等于做了相反操作。**32 次后 ft0 回到原值**，shuffle 等于没做 ❌

修复：让指令在 for 循环里只触发一次：

```c
case 0x0B:
    if (i == __builtin_ctz(current_mask)) {
        exec_vshfl(warp, current_mask, insn);
    }
    break;
```

PC 更新仍由外层 `lane->pc += 4` 处理，每个 lane 独立加 4，行为正确。

### 工具链问题：GAS 不认自定义助记符

标准 GAS 不知道 `vshfl.xor` 是什么。两条路：

- 用 `.insn r OPCODE, FUNCT3, FUNCT7, RD, RS1, RS2` 让汇编器按 R-type 拼机器码
- 用 `.word` 手编

选了第一条。但 `.insn r` 的 RS2 字段**只接受寄存器名**，不接受立即数。我们的 mask 是立即数，要绕：用 `x1` / `x2` / `x4` / `x8` / `x16` 等寄存器名 —— 它们的编号正好是 1 / 2 / 4 / 8 / 16，机器码层面等价于直接放 mask 值。

为了可读性写了宏：

```asm
.macro VSHFL_XOR rd, rs1, mask
    .insn r 0x0B, 0, 0, \rd, \rs1, x\mask
.endm

# 使用
VSHFL_XOR ft1, ft0, 1
```

`\mask` 展开为 `1`，再拼接成 `x1` 传给 `.insn r`，编出正确机器码。

### 使用约束

- **必须 32 lane 全 active**：mask < 32 就一定有伙伴，但伙伴未 active 时拿到的是 init 的 0，归约结果错
- **必须 fully converged**：在 if/else 内部用 shuffle 会拿到 path 残留的旧值

这两条约束跟早期 CUDA 一致（CUDA 9 之前的 `__shfl` 也要求 warp-uniform 上下文）。

### 验证

测试 kernel `shuffle.S`：32 个 lane 各加载 in[lane_id]（值为 1.0 ~ 32.0），做 5 步 xor 归约求和，每个 lane 写回 out[lane_id]。

期望：out 全部 32 个位置都等于 1+2+...+32 = **528.0**。

事先用 4-lane 简化模型手动推演验证 xor 归约语义：

| 步骤 | 状态（lane 0,1,2,3 的 ft0） |
|------|----------------------------|
| 初始 | 1, 2, 3, 4 |
| 第 1 轮 shfl(mask=1) → ft1 | 各 lane 拿到伙伴的旧值 |
| 第 1 轮 fadd | **3, 3, 7, 7** |
| 第 2 轮 shfl(mask=2) → ft1 | 跨距 2 的伙伴 |
| 第 2 轮 fadd | **10, 10, 10, 10** |

xor 归约的几个精妙之处：每个 lane 都拿到同一个最终结果（不需要广播）；xor 对称（"我读它"=" 它读我"）；32 个值 5 步收敛。

实测 32 lane PASS ✅

## 关键经验总结

### SIMT 调度

- **min-PC scheduling 是 IPDOM 栈的廉价替代**：实现简单，对结构化控制流（if/else）正确，对 data-dependent 循环不够鲁棒
- **PC 是天然的"栈"**：所有 lane PC 自然汇合时 current_mask 自动恢复，不需要显式管理
- **传 current_mask 而不是 active_mask**：当前 cycle 真正在执行的 lane，是更精确的语义边界
- **ebreak 清 active_mask 而不是 current_mask**：ebreak 让 lane 整个退出执行，不仅仅是这一 cycle 不参与

### Shuffle 实现

- **整个 warp 一次性处理**的指令，**绝对不能放进 per-lane 的 for 循环**：会被调用 32 次，rs1 == rd 时等于做了多次 shuffle
- **snapshot 必须先读后写**：保证读到的是旧值，避免读写竞争 —— 没有 snapshot 的话写早的 lane 会污染读晚的 lane
- **snapshot 读全 warp 但写回只对 current_mask**：兼顾"伙伴可能在另一 path"和"不污染另一 path"

### 工具链 / 编码

- **RV32 custom opcode 空间**：0x0B / 0x2B / 0x5B / 0x7B 是预留的，可放心占用
- **`.insn r` 不接受立即数当寄存器字段**：用 `x1`...`x16` 这样的寄存器名替代，机器码等价
- **`.macro` + `\arg` 拼接**：把丑陋的 `x\mask` 包装成可读的 `mask` 参数

### 调试

- **divergence 测试要双向检查**：不只验证"该写的写了"，还要验证"不该写的没写" —— 单向检查会漏掉 min-PC 调度把指令派给错误 lane 的 bug
- **shuffle 测试先在 4-lane 简化模型上手算一遍**：xor 归约的中间状态不直观，手算验证能在写代码前就 catch 掉语义错误

### 验证条件不要写错

- `if (a != X && b != Y)` 是"两个都错才报错"，常常是想表达 `||`（"任一错就报错"）。**今天这个 bug 出现在 test_divergence.c 的检查代码里**，虽然没造成测试假通过，但一旦真出问题，bug 会被掩盖

## 收益预览

新的 SIMT 基础设施直接支撑后续算子：

- **RMSNorm**：用 5 步 xor 归约算 ∑x²，替代之前考虑的"32 lane 冗余计算"方案，**性能提升 32×**
- **Softmax**：max 归约 + sum 归约，两次 xor 树形归约
- **Attention**：每个 head 内的 softmax 都受益
- **未来扩展**：加 `vshfl.up` / `vshfl.down` 是 trivial 的（funct3 不同分支），prefix sum 也能高效实现
