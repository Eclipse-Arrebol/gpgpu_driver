### Milestone 3.5 第二关 RoPE 详细成果

**RoPE kernel**(`kernels/rope.S`,**120 字节,30 条指令**),设计要点:

- **派发**:单 warp 32 thread,每 lane 一对(`lane_id = k`),严格匹配 head_dim/2 = 32 = lane 数
- **算法路径**:5 条 fp 指令(3×fmul + 1×fsub + 1×fmadd),因 `fnmsub.s` 在 device model 未实现,走"岔路 B"
- **循环不变量外提**:cos_k / sin_k 在 head 循环外查一次,所有 head 共享(只依赖 token_pos 和 lane_id,与 head 无关)
- **in-place 写回**:用 fp 寄存器暂存原 (x_lo, x_hi),先全部读完再开始写,数学依赖图先画清楚再写代码
- **args 接口最小化**:5 字段(x_off / sincos_off / num_heads / head_dim / token_pos),head_dim/2 在 kernel 内用 srli 算

**测试矩阵实测**(全 PASS,严格逐档执行,V18 教训坚持到位):

| 档             | num_heads | head_dim | token_pos | exact_match | max_err | 评价                  |
| -------------- | --------- | -------- | --------- | ----------- | ------- | --------------------- |
| baseline_pos0  | 1         | 64       | 0         | 64/64(bit)  | 0       | bit-perfect 恒等变换  |
| single_pos1    | 1         | 64       | 1         | 61/64       | 1.49e-8 | 1 ULP,首次真旋转      |
| two_heads_pos5 | 2         | 64       | 5         | 125/128     | 1.49e-8 | 1 ULP,验证 head 循环  |
| qwen_q_pos100  | 14        | 64       | 100       | 818/896     | 2.98e-8 | 2 ULP,Qwen Q 真实尺度 |

**1.49e-8 ≈ 2^-26,正好是 fp32 在 0.1~1 量级数据上的 1 ULP**——这是 fp32 的物理极限,不是 bug。`fmaf()` 强制单舍入策略起作用了。

### RoPE 设计决策(从 V18 草图到 V19 落地)

V18 §2 已完整设计,本次会话只做实施。设计稳定性证明了"先把数学依赖图画清楚再写代码"原则的价值——4 个决策(host 端 sincos 预计算 / Q 和 K 共享 kernel / 单 warp + head 循环 / in-place 寄存器暂存)实施过程中无一返工。

#### kernel 寄存器最终方案

| 寄存器        | 用途                                                         |
| ------------- | ------------------------------------------------------------ |
| `a0`          | kernel_args 指针(入口)                                       |
| `a1-a5`       | x_off / sincos_off / num_heads / head_dim / token_pos        |
| `s0`          | lane_id = k                                                  |
| `s1`          | h(head 循环计数器)                                           |
| `s2`          | head_dim_bytes = head_dim * 4(预算,head 跨步 + sincos 行步进复用) |
| `s3`          | k_byte_off_in_head = 2*k*4(预算)                             |
| `t0/t1`       | 临时地址计算                                                 |
| `fa0`         | cos_k(全程持有)                                              |
| `fa1`         | sin_k(全程持有)                                              |
| `fa2/fa3`     | x_lo / x_hi(per-iter)                                        |
| `ft0/ft1/ft2` | t1=x_lo*cos / t2=x_hi*sin / t3=x_hi*cos                      |
| `fa4/fa5`     | new_lo / new_hi                                              |

#### 巧妙的 stride 复用

`s2 = head_dim * 4` 既是 head 跨步(`base = x_off + h*s2 + s3`),也是 sincos 行步进(`sincos_addr = sincos_off + token_pos*s2 + s3`)。原因:`sincos 行 = head_dim/2 对 × 2 个 fp32 × 4 字节 = head_dim*4`,刚好等价。**省一条指令,代码更对称**。

### 本次会话隐性成就

1. **stub kernel 触发的诊断流程**:测试一开始 `rope.bin` 只有 5 条 args 加载指令(20 字节,可能是开发早期残留),`single_pos1` FAIL 时 `got = input` 信号一眼看出 kernel 没算。**好测试代码的核心价值——FAIL 时能给出精准信号**。
2. **baseline PASS 但实际是平凡通过**:第一次跑 baseline_pos0 PASS 了,但因为 stub kernel 啥也没改 in-place buffer,**baseline 也 PASS**。这印证了 V18 教训"baseline 单独 PASS 不能证明算法对"——必须 `single_pos1` 这种真旋转 case PASS 才算数。**未来设计测试矩阵时,baseline 的角色应清晰定位为"基础设施验证"而不是"算法验证"**。

## 4. 关键经验与原则

### 本次 milestone 3.5 第二关(RoPE)沉淀的新经验

#### `fmaf()` 强制单舍入是 host reference 与 device fmadd.s 严格平行的必要条件

host 端如果直接写 `a*b + c`,gcc `-O2 -ffp-contract=on`(默认)行为不确定:**可能合并为 fma(单舍入),也可能拆成 fmul + fadd(双舍入)**。device 的 `fmadd.s` 是 IEEE 754 单舍入,如果 host 拆成双舍入,**会产生 1~2 ULP 差异污染容差判据**——本来"对的算法"看起来像"略有偏差",难以区分是真 bug 还是 fp 行为不匹配。

**新原则**:涉及 fma 类操作时,host reference 必须显式调用 `fmaf(a, b, c)`(C99 标准库,glibc/musl 都按 IEEE 754 单舍入实现)。**不要相信编译器自动合并**。

```c
// ❌ 编译器行为不确定
float new_hi = x_lo * sin_k + t3;

// ✅ 严格单舍入,bit-equivalent 于 device fmadd.s
float new_hi = fmaf(x_lo, sin_k, t3);
```

3 条 fmul + 1 条 fsub 部分**不需要**fmaf,因为 mul 和 sub 之间编译器无合并空间(IEEE 标准不允许把 a*b - c*d 合并成单一 fma 反向操作)。**只有 fma 入口需要**。

#### baseline 单独 PASS 不能证明算法对——必须有"真正在算"的 case PASS

V18 §2 已经写了 "baseline 关键性:pos=0 时 cos=1, sin=0,RoPE 退化为恒等变换",并把 baseline 当作 "超强的算法对不对判据"。**这个表述需要修订**:

- baseline (token_pos=0) 真实的角色是 **基础设施验证**——证明 H2D / args / launch / D2H 链路通,数据没被神秘破坏
- baseline 的 bit-perfect **不能证明算法实现对**,因为 stub kernel(啥都不做)、空 kernel(只读不写)、错误 kernel(读完直接写回原值)**都能通过 baseline**
- 真正的 "算法对" 判据是**第二档及之后**——pos≠0 的 case 拿到非平凡的非零旋转结果,且 max_err 在 fp32 ULP 量级

本次会话开局 stub kernel 触发的剧本验证了这一点:`baseline_pos0 PASS` + `single_pos1 FAIL` 同时发生,FAIL 信号清楚指出 `got = input`。

**新原则**:测试矩阵中 baseline 的角色定位明确为 "infra check",非 baseline 第一档(本次的 single_pos1)是 "first algorithmic check"。**baseline PASS 不允许跳过非 baseline 第一档**——这是逐档原则在"什么算证据"层面的细化。

#### "好测试代码"的诊断价值在 FAIL 时才显现

stub kernel 时 `single_pos1 FAIL` 给的信号:

```
[0] got=-0.29999 ref=0.00620051 err=0.30619
[1] got=-0.19999 ref=-0.360488 err=0.160498
[2] got=-0.09999 ref=-0.0731756 err=0.0268144
```

**`got` 列就是 input 数据本身**——一眼看出 kernel 没动 buffer。这个诊断只用了 5 秒。

测试代码做对了三件事:

1. 三指标(exact_match + max_err + 头尾样本)
2. FAIL 时打印前 16 个样本(不只看 max_err)
3. bit pattern 对比(uint32_t 视角,不是 fp 比较)

这些都是 V16/V17/V18 累积的约定,本次再次验证价值。**未来新算子第一次跑 FAIL 时,优先检查测试输出能否一眼定位"got 是不是输入原值"——是 → kernel 没算;不是但和 ref 完全无关 → kernel 算错了别的东西**。

#### 数学依赖图先画清楚再写代码

V18 提出 "RoPE 设计前先画输入输出依赖图"。本次会话实施过程**无一返工**——4 个设计决策(host 预算 sincos / Q 和 K 共享 / 单 warp + head 循环 / in-place 寄存器暂存)直接落地。

特别是 in-place 安全性:**单 lane 内寄存器暂存 + 跨 lane 不重叠 + 跨 head 跨步 head_dim*4 不重叠**——这三层独立保证一起成立时,in-place 是安全的。**未来 in-place 算子的安全性论证应该走这三层结构**。

#### stride 复用(数值等价识别)

`head_dim * 4`(s2)同时是 head 跨步和 sincos 行步进。两者数值等价但语义不同——一个来自数据布局,一个来自 sincos 表布局。**识别这种数值等价能省指令、增对称性**。

#### 单 warp + head 循环模板适用范围

V18 vmul 已经验证 "单 warp 32 thread + stride-32 内部循环" 模板。本次 RoPE 是它的变体:**单 warp 32 thread + per-lane 一固定对 + head 循环**。两者本质相同——都是"用 32 lane 跑 N 次某个 32 宽度的工作"。

**未来 1D element-wise 类算子直接套这个模板**(包括将来可能写的 LayerNorm / activation 类)。GEMM 那种 "一线程一元素 + K 维累加" 是另一种模板,适合需要并行累加的场景。

## 5. 协作约定

### 沿用约定(V11~V18 已建立)

- **导师模式**:Claude 给设计方向、原理、code review;Eclipse 实际写代码
- **概念先行**:写代码前先把 "why" 讲清楚
- **review 节奏**:提交代码 → Claude 分别指出做对的部分和 bug → Eclipse 改 → 再提交
- **系统化 debug**:行为不符预期时,写 debug kernel dump 寄存器/内存状态
- **逐 phase 验证**:复杂算子写完一个 phase 立刻测,不要整个写完再 debug
- **测试输入要打破对称性**:周期/对齐输入会掩盖 reduce bug
- **检查循环坐标边界**:max_err = 0 + max_i = -1 看起来像没跑,需要 exact_match 计数消歧
- **每次会话末写 handover**(本文档为 V19)
- **汇编寄存器候选严格限定**:`t0-t6, s0-s11, a0-a7, ft0-ft11, fs0-fs11, fa0-fa7`
- **新汇编 kernel 的 VSHFL_XOR 宏一律用 `.word` 版本**,不用 `.insn r`
- **决策可以在实施中修订**:发现实施代价超预期立刻提出修订,但要走完整的"决策 → 实施"流程
- **多 buffer 上传 checklist**:每个 dispatch 前,host 输入 buffer 必须上传,device 中间 buffer 不应该被 host 数据覆盖
- **debug kernel 输出 ↔ host 读取严格对齐**:device 写 N 个 word,host 必须严格读 N 个 word
- **新算子先核对指令清单**:动手前列出该算子用到的所有 RV32 指令,grep device model 确认每条都已实现
- **每个 gpuMalloc 后立即检查返回值**(V16)
- **位拷贝类 kernel 优先 bit pattern 测试**(V16)
- **位拷贝类 kernel 优先 lw/sw 不走 flw/fsw**(V16)
- **报告强化**:验证段 max_err + exact_match + 头尾样本三指标(V16)
- **接口最小化**:派生量在 kernel 里算(V16)
- **接手既有代码先读源码确认**(V17)
- **fp 累加链路上 max_err = 0 必须解释**(V17)
- **新 host 测试模板默认动态读 kernel 大小**(V17)
- **1D kernel 派发统一用 `calc_dispatch_1d` 或固定 32 thread**(V17)
- **C 矩阵/输出 buffer 默认哨兵初始化 + 上传**(V17)
- **debug 报告打印 bit pattern 而不是只看 float**(V17)
- **打破对称偏移量必须 ≫ ULP**(V18)
- **新算子严格按"baseline → 第 2 档 → 真实尺度"逐档验证,不要跳档**(V18)
- **新 kernel 设计前先画"输入输出依赖图"**(V18)
- **跨 kernel 共享的有状态字段需明确约定 host 端状态机**(V18)

### V19 新增约定(本次)

- **涉及 fma 类操作的 host reference 必须显式 `fmaf()`**:不依赖编译器 `-ffp-contract` 行为。这是"reference 与 device 严格平行 fp 序列"原则的具体落实。
- **baseline (恒等变换 / 平凡输入) 的角色定位为基础设施验证,非"算法对"判据**:必须有非平凡第一档(real algorithmic check)PASS 才能称算法对。baseline PASS 不解锁跳过非 baseline 第一档。
- **in-place 算子安全性论证走三层结构**:(1) 单 lane 内寄存器暂存原值;(2) 跨 lane 内存区域不重叠;(3) 跨循环维度跨步不重叠。三层独立保证一起成立时 in-place 安全。
- **identify stride 等价性以省指令**:数据布局和辅助表布局可能巧合产生数值等价的 stride(如 RoPE 中 head_dim*4 同时是 head 跨步和 sincos 行步进),识别后单一寄存器复用,代码更对称。
- **stub kernel 触发的诊断剧本是好测试代码的核心价值体现**:测试代码 FAIL 输出能让人 5 秒内判断是 "kernel 没算 (got=input)" 还是 "kernel 算错了" (got 既不是 input 也不是 ref)。这是测试代码设计的隐性目标。

### Eclipse 风格备注

Eclipse 经常主动识别非显然问题(PC 饿死 corner case、SPI opcode sniffing 缺陷、snapshot 语义、VSHFL_XOR 串联怀疑、elems_per_lane 是否冗余、in-place args buffer 是否会被覆盖等),提精确追问。Claude 应当把 Eclipse 的怀疑当作严肃的输入对待——经常方向是对的。

V19 中 Eclipse 全程问题精准:从 args 设计、lane 分工、in-place 写回安全性、host buffer 数量,每一个问题都直接命中关键设计点;在"框架太复杂"时主动要求 Claude 给完整代码学习(高效切换模式),而不是僵硬走"我写,你 review"流程。**这种灵活切换是高效协作的关键**。

### SIMT 模型

- mhartid CSR 位域:`[31:13] block_id, [12:5] warp_id, [4:0] tid`
- lane id 取法:`csrr s0, mhartid; andi s0, s0, 0x1f`
- single-warp kernel 不需要检查 warp_id

### Dispatch 语义

- `block_size = bx * by * bz`,kernel 看到 1D 线性 thread id
- `num_warps = ceil(block_size / 32)`,block 内总线程数不必 32 倍数
- grid 多 block 顺序执行,不提供并行性 → 统一 `block = {total, 1, 1}, grid = {1, 1, 1}`
- `kernel_args` 经 `gpr[10]`(即 `a0`)传给所有 lane
- `gpgpu_core_exec_warp` 单 warp **100000 cycle 上限**(硬编码,记债务)
