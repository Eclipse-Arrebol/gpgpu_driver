### 2. qkt_scale.S 设计与实现

**算法**:`S[i,j] = scale * Σ_k Q[i,k] * K[j,k]`,scale = 1/sqrt(64) = 0.125

**并行策略**:32 lane,每 lane 算 8 个输出

- `i = lane_id >> 1`(0..15)
- `j_base = (lane_id & 1) << 3`(0 或 8)
- 每 lane 内层 8 个 local_k × 64 次 fmadd = 512 fmadd / lane

**偏移化简**:`S_lane_base = S_base + lane_id * 32`(代数巧合,8 elem × 4 byte × i_step + j_step 对齐到 32)

**关键实现细节**:

- t0 在 setup 后被复用为 local_k 计数器(lane_id 派生量已固化到 t1/t2/t4)
- scale 通过 `fmv.s.x fs0, a4` 从整数寄存器搬到浮点寄存器(关键:`fmul.s ft2, ft2, a4` 这种"浮点指令传整数寄存器"GAS 不会报错,会编码出错误指令读未初始化浮点寄存器)
- 内层循环 `li a7, 64` 没提到外层(性能债,不影响正确性)

**测试结果**:max_err 在 1e-7 量级,4 个 spot check 全部精确匹配。

### 3. softmax16.S 设计与实现

**算法**:对 S [16, 16] 每行独立 softmax → P [16, 16]

**并行策略**:**首次使用 grid 维度**——grid=16 (每 block 处理 1 行),block=32 (1 warp)

- `block_id = mhartid >> 13`(从 V13 沉淀的 mhartid 编码 `(block<<13) | (warp<<5) | thread` 反推)
- `lane_id = mhartid & 0x1F`
- 每 block 内,lane 0..15 各处理一个真实元素,lane 16..31 填 **-INF 哨兵**

**-INF 哨兵的妙处**:

- max reduce:`fmax(real, -INF) = real`,虚 lane 不影响 max
- sum reduce:`exp(-INF) = 0`,虚 lane 不影响 sum
- **不需要任何 mask 逻辑**,32 lane 统一跑 5 步 XOR 蝶式 reduce

**寄存器流分析**(细致复杂但无冲突):

- ft0:工作值(x → max → exp → e/sum)
- ft1:-INF 哨兵(setup 设置后不变,只供虚 lane 用作 fmv 源)
- ft2:e_local 备份(B reduce 前保存,C divide 时用)
- ft3:reduce 临时 → global_max(覆盖)
- ft4:reduce 临时 → global_sum(覆盖)

**测试结果**:max_err = 5.96e-8,接近 float ULP 物理下限。

### 4. pv.S 设计与实现

**算法**:`O[i,k] = Σ_j P[i,j] * V[j,k]`,P [16,16] @ V [16,64] → O [16,64]

**并行策略**:32 lane,每 lane 算 32 个输出元素(类比 qkt_scale)

- `i = lane_id >> 1`,`k_base = (lane_id & 1) << 5`(0 或 32)
- 每 lane:外层 32 个 local_k × 内层 16 次 fmadd

**偏移化简**:`O_lane_base = O_base + lane_id * 128`(同 qkt_scale 套路,代数巧合)

**关键实现细节**:

- V 跨行 stride = head_dim × 4 = 256 字节,用 `addi a6, a6, 256` 推进
- 没有 scale,比 qkt_scale 少一条 fmul

**测试结果**:**max_err 严格 0**——和 host reference bit-exact。原因:gcc 默认开 fp-contract,把 host 的 `acc += P[i] * V[j]` 编译成 fmadd,和 device fmadd.s 完全等价路径。

### 5. test_attention.c 端到端

**3 个 kernel 一次性上传,顺序 dispatch,中间结果不出 device**:

```
Phase 1: gpuInit
Phase 2-3: 3 个 kernel 各 fopen + 各 gpuMalloc + 各上传
Phase 4: 6 个 data buffer 分配 (Q, K, V, S, P, O)
Phase 5: 3 个 args buffer 分配
Phase 6: 输入填充 + host reference 计算(三段 attention)
Phase 7: 输入数据上传(Q, K, V)
Phase 8: 3 个 args 上传
Phase 9: 顺序 dispatch(qkt grid=1 → softmax grid=16 → pv grid=1)
Phase 10: 仅读回 O
Phase 11: 验证
```

**关键概念**:S_off 和 P_off 在 args 之间共用——qkt 写 S,sm 读 S 写 P,pv 读 P 写 O。中间结果不读回 host,这是标准 GPU 编程模式。

**测试结果**:

```
max_err = 8.94e-08 at (14, 24), fail = 0
O[0,0]   = 0.259232 (ref 0.259232)
O[0,63]  = 0.309632 (ref 0.309632)
O[8,32]  = 0.292973 (ref 0.292973)
O[15,63] = 0.324855 (ref 0.324855)
```

8.94e-8 ≈ 2^-23.4,等于一次 fdiv 的 ULP——三个 kernel 串联后总误差仅 1 个 ULP,达到浮点物理下限。

## 关键经验沉淀(累积到 V13 之上)

### 关于决策迭代

V13 沉淀强调"决策对齐后再写代码",但**决策本身可以在执行中迭代**——决策 3 从 "pad-to-32 复用" 改为 "新写 softmax16.S grid=16",原因是实施时发现复用代价大于重写。**沉淀的原则**:

- 决策对齐是为了避免无方向的反复改写,不是为了固守初版
- 实施中发现决策代价超预期 → 立刻提出修订,等价方案选最简单的
- 修订决策要重新走一遍"决策 → 实施"流程,不要在执行中悄悄偏移

### 关于边界检查的取舍

Step 3(lane load/store 检查)搁置的判断:**接口扩展的成本超过当前阶段的收益**。沉淀:

- 防御性工程的优先级是**变量**,随项目阶段调整
- attention 还没收尾就投入 Step 3 接口重构 → 上下文切换成本高
- 真出 segfault 时,V13 的"写 debug kernel dump 寄存器"方法仍然能定位
- 风险记账(工程债务表)+ 阶段性专注 > 完美主义

### 关于"max_err = 0 不一定是好消息"

pv.c 第一版打印 `max_err = 0.000e+00 at (-1, -1)`——max_i / max_k 仍是初始 -1,**说明检查循环从未触发更新**,原因是循环边界写错只检查了 256/1024 元素,且这部分 host 和 device 浮点路径恰好 bit-exact。

**沉淀的检查模式**:

- max_err 数值看起来好,要看 max_i/max_k 是否 -1(初始值)
- 如果是 -1 → 检查循环范围是不是写对了
- 如果是真实坐标 + max_err = 0 → 大概率是 host gcc 用 fmadd 让两端 bit-exact,**这是真 PASS**
- spot check 4 个角永远是必要的——它独立于 max_err 逻辑,不依赖循环正确

### 关于 V 漏上传陷阱

test_attention.c 第一版**漏了 `gpuMemcpy(V_off, V, ...)`**,且**多了一个 `gpuMemcpy(P_off, P, ...)`**(把脏数据上传到中间 buffer)。后者被 softmax 写入覆盖侥幸不影响结果,前者会让 pv 读到全 0 的 V 算出错误 O。

**沉淀的检查清单**:dispatch 前,**所有 host-only 输入 buffer 必须有对应的 gpuMemcpy 上传**;**所有 device 中间 buffer 不应该被 host 数据上传**(除非显式做 init-to-0)。Attention 三个 kernel 是这个 pattern 的最小实例,Qwen2 端到端会有几十个 buffer,需要更系统的 checklist。

### 关于 mhartid 三段编码

V13 沉淀里只记录了 mhartid 编码格式 `(block<<13) | (warp<<5) | thread`,本次会话**首次在 kernel 内反向解码 + 用 grid 维度**。验证:

- `srli t0, t0, 13` 提 block_id ✓
- `andi t0, t0, 0x1F` 提 lane_id ✓(0x1F mask 掉所有高位,即使 block>0 也安全)
- 旧 kernel(grid=1)的 `andi t0, 0x1F` 写法在新 grid>1 配置下不会受影响——**block_id 不需要的 kernel 完全不用改**

### 关于 GAS 的 t7..t9

V13 已记"整数寄存器只到 t6",这次**再次踩中**——写 softmax16.S 时设计文档用了 `li t9, 16`,build 失败。**沉淀升级**:

- 给 Eclipse 写汇编时,候选寄存器名严格限定:`t0-t6, s0-s11, a0-a7, ft0-ft11, fs0-fs11, fa0-fa7`
- t7-t9 / fa8+ / fs12+ 这些 GAS 不认的名字**永远不出现在伪代码或设计中**
- "illegal operands" 错误优先怀疑寄存器名不存在,其次怀疑 funct7 编码

### 关于 VSHFL_XOR 宏的 `.word` 化

V13 沉淀已记"长期方案:全部用 .word 拼"。本次再次踩到——softmax16.S 用了 `.insn r` 版本的 VSHFL_XOR,在该 GAS 版本下报 "illegal operands"。**沉淀升级**:

- 新 kernel 一律用 `.word` 拼 VSHFL_XOR(已在 softmax16.S 落地)
- 现有 rmsnorm.S / softmax.S 的 `.insn r` 版仍是 latent bug,工程债务记录,以后统一改

### 关于"忽略掉的边界 case"

qkt_scale 内层 64 次 + softmax16 5 步 reduce + pv 16 次内层——这些循环边界都是**精确等于**的(`bne` + 比较常量),没有 `<=` / `<` 边界混淆。沉淀:**循环计数从 0 起步、`bne ctr, limit` 退出**是 V13 sotmax 假阳性后的标准模式,这次三个 kernel 都正确执行了这个模式。

### 关于"bit-exact 不是 bug 而是真相"

pv 测试 max_err = 0,验证了 4 个 spot 后判定真 PASS。原因是 host gcc `-O` 默认开 `-ffp-contract=fast`,`acc += a*b` 自动合并成 fmadd,和 device fmadd.s bit-equal。沉淀:

- 单 kernel bit-exact 是常见现象(host 和 device 都用 fma)
- 多 kernel 串联中,如果有 `exp` / `fdiv` / `fsqrt` 等非 fma 操作,bit-exact 不可能,会出现 ULP 级误差
- max_err 在 1e-7 ~ 1e-9 区间是健康信号,意味着没有大 bug 也没有完全 bit-exact 的"错觉式 PASS"

## 下一步计划

### 短期:Embedding lookup + argmax(milestone 3.3 / 3.4)

**argmax**(简单)

- 输入:logits 向量(长度 = vocab_size,Qwen2 是 151936)
- 输出:argmax 的 token id(uint32)
- 算法:warp 内 reduce max + 同步追踪 index
- 复杂度:1 warp,处理一个长向量

**Embedding lookup**(中等)

- 输入:token id(uint32)、embedding table([vocab_size, hidden_dim])
- 输出:hidden_dim 维向量
- 算法:gather 操作,从 table 第 token_id 行复制 hidden_dim 个 float
- 复杂度:1 warp × 32 lane,每 lane 复制 hidden_dim/32 个元素(896/32=28)

### 中期:milestone 3.5 端到端 Qwen2-0.5B 推理

**Transformer block 串联**:

```
input_token_id
    ↓ embedding lookup → hidden [896]
    ↓ ┌─────────── 24 个 layer 循环 ───────────┐
    ↓ │  RMSNorm
    ↓ │  Linear(QKV projection)  ← 需要 GEMM 接 [896]→[896*1.5] (Q,K,V combined for GQA)
    ↓ │  Attention(Q,K,V → O)
    ↓ │  Linear(O projection)    ← GEMM
    ↓ │  Residual add
    ↓ │  RMSNorm
    ↓ │  Linear(MLP up + gate)   ← GEMM
    ↓ │  SiLU + multiply (gating)
    ↓ │  Linear(MLP down)        ← GEMM
    ↓ │  Residual add
    ↓ └────────────────────────────────────────┘
    ↓ RMSNorm (final)
    ↓ Linear(LM head)             ← GEMM
    ↓ argmax → next_token_id
```

**所需新算子**:

- Residual add(element-wise vector add)
- Element-wise multiply(SiLU 之后的 gating)
- 通用 GEMM(现有 GEMM 是固定 shape,需要参数化)

**所需基础设施**:

- 模型权重加载(从 GGUF / safetensors 读到 host,再上传到 VRAM)
- INT8 量化(规划但优先级低,可以先 float32 跑通)
- 推理循环 host 端编排
