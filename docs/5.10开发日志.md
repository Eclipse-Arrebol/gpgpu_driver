| #    | 决策                              | 选定方案                                                |
| ---- | --------------------------------- | ------------------------------------------------------- |
| 1    | KV cache layout(下一个 milestone) | `[layer][kv_head][seq][head_dim]`,读连续                |

| 9    | milestone 切块                    | 先 decode 三件套,KV cache 状态机推后                    |
| 10   | 测试矩阵                          | 4 档(baseline / S<max_seq / GQA / Qwen2 真实尺度)       |
| 11   | 诊断方式                          | 失败时抽检 scores 和 P 中间结果                         |

### 关键设计决策的论据(以备未来回顾)

**为什么走路径 C 而非 B(参数化重写老三件套)?**

- 老三件套 lane 分工是 seq_len=16 量身定做,通用化等于完全重写
- 重写老 kernel 还要兼容 prefill 和 decode 两种语义,负担更重
- decode kernel 数学退化(Q 单行)结构更简单
- prefill 阶段先用"循环 N 次 decode"顶上,N 通常 < 100 可接受

**为什么拆法 β(3 个独立 kernel)而非 γ(qkt+softmax fused)?**

- **关键认知**:RV32 没有动态寄存器索引能力,fused 想"scores 留寄存器"必须循环展开,但 cur_pos 运行时变化无法编译时展开 → fused 优势消失
- 一旦 scores 必须落 VRAM,fused 和不 fused 几乎等价
- β 的三件套结构清晰、debug 边界明确

## 4. pv_decode.S 设计(下次开工)

### 数学

```
对每个 query_head qh ∈ [0, num_q_heads):
  kh = qh / q_per_kv
  for k in 0..head_dim:
    O[qh, k] = sum_j(P[qh, j] * V_cache[kh, j, k])    j ∈ [0, S)
```

### lane 分工

- `grid={num_q_heads, 1, 1}`,每 block 处理一个 query_head
- 32 lane 沿 k(head_dim)切,每 lane 算 `head_dim / 32` 个 k(head_dim=64 时为 2)
- 每 lane 内部对 j ∈ [0, S) 串行 reduce
- **没有跨 lane reduce**

### args(7 个字段,**没有 scale**)

```c
struct pv_decode_args {
    uint32_t P_off;          /* [num_q_heads, max_seq] */
    uint32_t V_cache_off;    /* [num_kv_heads, max_seq, head_dim] */
    uint32_t O_off;          /* [num_q_heads, head_dim] */
    uint32_t cur_pos;
    uint32_t max_seq;
    uint32_t head_dim;
    uint32_t q_per_kv;
};
```

### 期望

由于无跨 lane reduce,**理论 max_err 也应该 = 0**。这是项目里第二个"max_err=0 类"kernel,可作为 softmax_decode 写完后判断"reduce 误差是否符合预期"的对比基准。

## 5. softmax_decode.S 设计(留到 pv_decode 之后)

### 数学(in-place,3 遍扫)

```
对每个 query_head qh ∈ [0, num_q_heads):
  /* 第 1 遍:求 max */
  m = max(scores[qh, 0..S-1])

  /* 第 2 遍:算 e[j] = exp(scores[j] - m),原地写回,累计 sum */
  for j in 0..S:
    e = exp(scores[qh, j] - m)
    scores[qh, j] = e            ← in-place 写
  sum = sum(e[0..S-1])

  /* 第 3 遍:除 sum,原地写回 */
  for j in 0..S:
    scores[qh, j] = scores[qh, j] / sum    ← in-place 写
```

### lane 分工

- 同 qkt_decode:stride=32 沿 j
- 跨 lane reduce 2 次:max-reduce(VSHFL_XOR 5 步)+ sum-reduce(VSHFL_XOR 5 步)
- 模板对照:**rmsnorm.S**(local sum-of-squares + 5 步 XOR shuffle)

### args(3 个字段)

```c
struct softmax_decode_args {
    uint32_t scores_off;     /* in-place buffer */
    uint32_t cur_pos;
    uint32_t max_seq;
};
```

### 期望

由于跨 lane reduce 改变累加顺序,**max_err 不可能 = 0,会有 ULP 级误差**(预期 1~2 ULP,类似 RoPE)。

### corner case 提醒

- **S=1**(decode 第一个 token):lane 0 持有 1 元素,其他 lane 走"虚 lane"(local_max=-INF, local_sum=0 作为 reduce 中性元)。**结构上自然处理,不需特殊代码**
- **第二遍写 e[j] 后,跨 lane reduce sum 时,read 到 scores 还是 e?** —— 各 lane 持有自己的 local_sum 寄存器,不再读 scores buffer。安全。

### in-place 三层安全性论证(沿用 V19 RoPE 的模式)

- **单 lane 内**:每次先 flw 到寄存器再算再 fsw,寄存器暂存 ✓
- **跨 lane**:每 lane 负责的 j 不重叠 ✓
- **跨循环维度**:三遍扫都是一维 j 维度,无嵌套维度交叉 ✓

## 8. 关键学习与原则(沉淀)

### 8.1 新增原则

**原则:SIMT kernel 设计时,必须先把目标 ISA 的寄存器索引能力摆出来**

RV32 没有 array-of-registers 能力(寄存器名编译时确定),所以"留寄存器"只对编译时已知数量的临时值可行;运行时长度的中间数组**必然落 VRAM**。这条限制让"fused = 零中间内存"在 RV32 上是个幻觉。

是 V14+"接口最小化"和 V11+"先把数学依赖图画清楚再写代码"的延伸——**还要把"硬件支持的数据存放方式"画清楚**。

**原则:接手既有 kernel 时必须读源码再下结论**

本次会话曾因看走眼 `qkt_scale.S` 的 `lane_id >> 1` 范围,误判"qkt_scale Q=4行 vs pv P=16行 不匹配"。读 SIMT kernel 时 **lane_id ∈ [0, 32)** 是基本常识,不应糊涂。Eclipse 的"我还有问题/疑虑"按住推进、要求贴源码核对,正是 V17 接手既有代码原则的兑现。

**原则:max_err=0 适合做"对比基准",不是常态**

单 lane 内 reduce + 累加顺序一致 → 可达 max_err=0。任何走跨 lane reduce 的 kernel(softmax、rmsnorm)期望 1~2 ULP 误差。判断"误差是否合理"不再靠经验,而靠**结构性预测**。

**原则:毒数据是 KV cache 时代验证 cur_pos 边界的核心工具**

case 1 把 K_cache 后半填 999.0 验证 scores 不被污染——比"只测正确数据"强得多。后续 pv_decode、softmax_decode 测试都应该用同样思路。

### 8.2 流程经验

**6 层设计推演 + 选择题对齐 → 汇编一遍过**

本次 qkt_decode 推演了 6 个层次(数学 → 拆 kernel → lane 分工 → buffer → host 接口 → 测试),每个决策点用单选题锁定。汇编代码第一遍写完,bug 全是机械式细节(args offset 错位、读 hartid 多次),没有结构性 bug。

**这是项目原则"先把 why 讲清楚再写代码"的标志性兑现**。沉淀格式为后续 milestone 模板:

```
层次 0:数学梳理(到底要算什么)
层次 1:kernel 拆分(几个 kernel,fused 与否)
层次 2:每 kernel 的 lane 分工和 reduce 策略
层次 3:host 接口和 buffer 生命周期
层次 4:测试矩阵(逐档放大,什么 bug 能在哪档暴露)
↓
开始写汇编
```
