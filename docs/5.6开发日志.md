### Milestone 3.4 详细成果

**完整的 embedding lookup kernel**(`kernels/embedding.S`),设计要点:
- 32 lanes,lane `i` 处理 `d ∈ {i, i+32, i+64, ...}`(stride-32 交错布局,coalesced access 友好)
- 无 reduce、无跨 lane 通信、无 chunk 边界——纯位拷贝
- token_id 通过 device buffer 接口传入(不是 kernel arg 标量),为后续接 argmax 输出做准备

### 519 MB alloc 失败的解决方向(供 3.5 开局讨论)

- **选项 A:扩 VRAM 到 2GB**——根本解,但要动 device model 和 PCI BAR,需要确认 QEMU virt machine 撑得住
- **选项 B:改 buddy 策略,允许非 2 的幂大小**——本质是放弃纯 buddy 改 hybrid(buddy + 直接管理大块)
- **选项 C:大 tensor 走专用大 chunk allocator**——slab/buddy 之外加第三层"giant chunk" pool,大 alloc 直接从底层 VRAM 头部连续切
- **选项 D:权重 tensor 不全驻留 VRAM,推理时按 layer 动态 swap**——架构层面最复杂但最贴近真实硬件做法

倾向选项 C(实现简单,语义清晰),但要看 3.5 端到端的具体内存峰值再定。

## 4. 关键经验与原则

### 本次 milestone 3.4 沉淀的新经验

#### bit pattern 测试方法 vs fp 算术测试

embedding lookup 在算法层面是位拷贝,fp 语义无关。测试数据用 `(t << 16) | d` 这种 bit pattern,host expected 和 device 输出 bit-for-bit 严格相等,exact match 必须 100%。

**优势**:
- 完全避开 fp 精度争议(`(float)151935000` 在 fp32 里是不精确的,但 bit pattern 永远精确)
- 任何一位错都立即暴露
- 同时兼容 t=0 和 t=2047 边界
- 写得出的 expected 算法极简(一行 `(TOKEN_ID << 16) | d`)

**劣势**:
- 打印出来 `O[d]=2.43e-39` 之类的非正常 float 值,看着难受,但 raw_u32 一目了然

**适用范围**:位拷贝类 kernel(embedding、reshape、permute、cast 不变位)。涉及 fp 算术的 kernel(rmsnorm、softmax、gemm)继续用 fp 数据 + 容差比对。

#### `max_err = 0` + `max_d = -1` 的报告歧义

V15 §5 教训"`max_err = 0` + `max_i = -1` 是循环没跑的信号"——但本次 mini-1/mini-2/real 三次都看到这个报告,实际是"全部 bit-perfect 匹配 → diff 永远是 0 → `if (diff > max_err)` 永远不进 → max_d 保持初值 -1"。

**消歧办法**:验证段除了 max_err,**额外打印 `exact_match` 计数和头/尾样本**:
```c
int exact_match = 0;
for (uint32_t d = 0; d < D; d++) {
    if (O_u32[d] == expected_u32[d]) exact_match++;
}
printf("exact match: %d / %u\n", exact_match, D);
```
看到 `exact match: 896 / 896` 才算真 PASS,只看 `max_err = 0` 不能下结论。

进一步的报告强化:
```c
int   max_d   = 0;        // 不要初始化为 -1
float max_err = -1.0f;    // 哨兵值
// ... 循环 ...
if (max_err < 0) {
    printf("WARNING: validation loop did not execute\n");
}
```
这样真正的"循环没跑"和"全对"在报告里能严格区分。

#### `gpuMalloc` 失败必须立即检查返回值

本次 V=151936 测试踩坑:`gpuMalloc(519MB)` 因 buddy 没空间返回 `(uint64_t)-1`,host 代码没检查就继续 `gpuMemcpy` / kernel launch,看到的现象是"device 一行都没写"——非常容易误判为 kernel bug。

**铁律**:每个 `gpuMalloc` 后立即检查:
```c
uint64_t off = gpuMalloc(ctx, size);
if (off == (uint64_t)-1) {
    fprintf(stderr, "gpuMalloc(%zu) failed\n", size);
    return 1;
}
```
没有这层检查,buddy/slab 失败会级联成"看起来像 kernel bug"的假象,debug 路径完全偏离。

#### buddy allocator 在"刚过 2^N"尺寸的恶劣行为

519 MB 是 buddy allocator 的恶劣点:向上取整到 2^30 = 1 GB,而 VRAM 总量也是 1 GB,且头部已被其他小 alloc 切割,1 GB 连续块拿不出来。

**这不是 buddy 实现 bug,是 2 的幂粒度的固有代价**。任何接近但超过 2^N 的尺寸都会浪费一倍空间(申请 X,实际占 2X)。Qwen2-0.5B 的所有大权重 tensor(embedding 519 MB、QKV weights 几十 MB、MLP weights 上百 MB)都会反复踩这个坑。

**测试侧应对**:为了算法层验收,可以缩小 V 让 alloc 装得下(V=2048 验证了 mul 中等大值正确,等价于"在 buddy 装得下的前提下 kernel 在 D=896 主循环 28 轮下正确")。

**生产侧应对**:进 milestone 3.5 时必须解决,见 §3 工程债务。

#### 整数通过 lw/sw 直接搬,不走 fp 通道

argmax 那次为了 reduce 把整数 idx 通过 fmv.s.x 塞进 fpr 通道做 vshfl。本次 embedding 是位拷贝,**完全不需要这个 trick**——直接 `lw`/`sw` bit pattern。即使 table 存的是 fp32,lw/sw 也是位级别操作,不解释 IEEE 语义,不触发 NaN-boxing。

这其实是 RISC-V ISA 设计的优雅之处:整数 load/store 对 fp 数据天然安全,只要你不想做 fp 算术。**写位拷贝 kernel 时优先 lw/sw,不走 flw/fsw**。

### 之前(V11–V15)的核心经验仍然有效

- **越界 host buffer 读会伪造 device bug**:debug kernel 输出大小 = host buffer 大小 = host 读取大小,严格相等;哨兵 0xDEADBEEF 初始化是标准防御
- **每加一个新算子,先核对指令清单**:V15 fmv.x.s 漏实现是教训源头;本次 mul/csrr 等都核对过,无 surprise
- **整数通过 float 通道搬运的设计模式**(argmax 用):仍是 reduce 类 kernel 处理 "value + 元数据" 配对的标准模板
- **Latent bugs masked by coincidence**:kernel_args 偶然对齐、fmv.x.s 之前 kernel 凑巧不用——本次 mul 在大值下也是这种"早期 kernel 用不到"的指令,提前覆盖
- **Slab routing 必须严格**:`<` vs `<=` 一字之差,silent memory overwrite
- **Warp shuffle 放置**:warp-wide 操作放进 per-lane 循环会执行 N 次
- **SIMT divergence 取舍**:Min-PC 调度处理发散正确但下 PC 循环可能饿死 lane
- **GNU 汇编器约束**:新约定一律 `.word` 直接编码
- **FP 算术需要 type-punning**:对 `uint32_t` 位模式直接做算术运算是 recurring bug class
- **大 struct 在堆上**:`gpgpu_ctx` 必须 heap-allocate
- **buddy split 正确性**:左半块必须用本地指针保存
- **MSI-X 在 SiFive PLIC 不可用**:必须用 Legacy INTx
- **`qemu/osdep.h` 第一个 include**

## 5. 协作约定

### 沿用约定

- **导师模式**:Claude 给设计方向、原理、code review;Eclipse 实际写代码
- **概念先行**:写代码前先把 "why" 讲清楚
- **review 节奏**:提交代码 → Claude 分别指出做对的部分和 bug → Eclipse 改 → 再提交
- **系统化 debug**:行为不符预期时,写 debug kernel dump 寄存器/内存状态,不靠猜
- **逐 phase 验证**:复杂算子写完一个 phase 立刻测,不要整个写完再 debug
- **测试输入要打破对称性**:周期/对齐输入会掩盖 reduce bug
- **检查循环坐标边界**:`max_err = 0` + `max_i = -1` 看起来像没跑,需要 exact_match 计数消歧
- **每次会话末写 handover**(本文档为 V16)
- **汇编寄存器候选严格限定**:`t0-t6, s0-s11, a0-a7, ft0-ft11, fs0-fs11, fa0-fa7`
- **新汇编 kernel 的 VSHFL_XOR 宏一律用 `.word` 版本**,不用 `.insn r`
- **决策可以在实施中修订**:发现实施代价超预期立刻提出修订,但要走完整的"决策 → 实施"流程
- **多 buffer 上传 checklist**:每个 dispatch 前,host 输入 buffer 必须上传,device 中间 buffer 不应该被 host 数据覆盖
- **debug kernel 输出 ↔ host 读取严格对齐**:device 写 N 个 word,host 必须严格读 N 个 word(不多不少)
- **新算子先核对指令清单**:动手前列出该算子用到的所有 RV32 指令,grep device model 确认每条都已实现

### V16 新增约定(本次)

- **每个 `gpuMalloc` 后立即检查返回值**:不检查会让 buddy/slab 失败级联成"看起来像 kernel bug"的假象,严重误导 debug 路径
- **位拷贝类 kernel 优先 bit pattern 测试**:数据用 `(t << 16) | d` 这种 bit pattern,host expected 和 device 输出 bit-for-bit 比对,完全避开 fp 精度争议
- **位拷贝类 kernel 优先 lw/sw 不走 flw/fsw**:RISC-V 整数 load/store 对 fp 数据天然安全,只要不做 fp 算术。少一个 NaN-boxing 风险
- **报告强化**:验证段除了 max_err,必须额外打印 `exact_match` 计数 + 头尾样本,消除"全对"和"循环没跑"的报告歧义
- **接口最小化原则**:args struct 字段数尽量少,派生量(如 `elems_per_lane = D/32`)在 kernel 里计算,不让 host 重复传。理由:host/device 不一致风险 > 多一条 srli 的开销

### Eclipse 风格备注

Eclipse 经常主动识别非显然问题(PC 饿死 corner case、SPI opcode sniffing 缺陷、snapshot 语义、VSHFL_XOR 串联怀疑、本次的 elems_per_lane 是否冗余等),提精确追问。Claude 应当把 Eclipse 的怀疑当作严肃的输入对待——经常方向是对的。

本次 Eclipse 主动问 "elems_per_lane 这个的作用是" 直接触发了"接口最小化"的讨论和方案 B 的采纳,是对的方向。
