### 0\. 先建立共同语境:KV cache 到底是什么、为什么要有

Qwen2-0.5B 是 decoder-only transformer,生成第 t 个 token 时:

- 当前 token 经过 Q/K/V projection → 产生 `Q_t, K_t, V_t` (每个都是 `[num_heads, head_dim]`)
- attention 阶段需要 `softmax(Q_t · [K_0, K_1, ..., K_t]^T / sqrt(d)) · [V_0, V_1, ..., V_t]`
- **关键观察**:`K_0 ... K_{t-1}` 和 `V_0 ... V_{t-1}` 在生成第 0..t-1 个 token 时 **已经算过了**,而且永远不会再变(权重不变 + 输入不变)

如果每次都重算 → O(t²) 计算量,生成长序列会爆炸。

**KV cache = 把历史的 K\_0..K\_{t-1}, V\_0..V\_{t-1} 保存在 VRAM 里,每生成一个新 token,只算 K\_t/V\_t 并 append 进去** 。

所以 KV cache 不是一个"算子",而是:

- **VRAM 上的一块预分配 buffer** (layer × seq\_len × kv\_heads × head\_dim × 4 bytes)
- **host 侧的一套状态机**,跟踪 "下一个 token 该写到 cache 的哪一行"
- **attention kernel 的读取路径**,从 cache 读 `[0..cur_pos]` 这段 prefix(而不是完整矩阵)

---

### 1\. 第一个设计决策:cache 在 device 侧的 layout(影响后续所有 attention 调用)

KV cache 的物理布局有两种主流选项,各有取舍。这是 **最关键的决策**,因为 attention kernel 的读 stride 完全由它决定。

#### 选项 X:\[layer\]\[seq\]\[kv\_head\]\[head\_dim\](seq 在外,head 在内)

```
对单个 layer 的 K cache:
  位置 (s, h, d) 的偏移 = s * (kv_heads * head_dim) + h * head_dim + d
  
写入第 t 个 token 时:
  - 一次写一整行 [kv_heads][head_dim],连续内存
  - dst_addr = base + t * (kv_heads * head_dim * 4)
  
attention 读取(给定 head h,读历史所有 K_h):
  - 第 s 个 token 的 K_h 在 base + s*(kv_heads*head_dim*4) + h*(head_dim*4)
  - 步长 = kv_heads * head_dim * 4(跳跃式访问)
```

**特点**:**写入连续** (快),**读取跳跃** (慢)。

#### 选项 Y:\[layer\]\[kv\_head\]\[seq\]\[head\_dim\](head 在外,seq 在内)

```
对单个 layer 的 K cache:
  位置 (h, s, d) 的偏移 = h * (max_seq * head_dim) + s * head_dim + d

写入第 t 个 token 时:
  - 每个 head 单独写一段 head_dim,不连续
  - dst_addr_for_head_h = base + h*max_seq*head_dim*4 + t*head_dim*4
  
attention 读取(给定 head h,读历史所有 K_h):
  - 第 0..cur_pos 个 token 的 K_h 完全连续
  - 步长 = head_dim * 4(连续读)
```

**特点**:**写入跳跃** (慢),**读取连续** (快)。

---

#### 怎么选?

关键是看 **"写 vs 读" 的频率比**:

- **每个 token 写 1 次,但 attention 读会被 dispatch `num_query_heads` 次** (Qwen2-0.5B GQA: query\_heads=14, kv\_heads=2,每个 kv\_head 被 7 个 query\_head 共享读)
- 即使是同一个 kv\_head,attention 阶段也要从头到尾扫一遍 cache
- **读频率远 > 写频率**

工业界主流是选项 Y(head 在外),但需要 **写入时多次 dispatch 或者用 stride store** 。

不过对我们来说,有个 **额外考虑**:我们的 `pv.S` 和 `qkt_scale.S` kernel 已经写好了,它们的输入布局假设是什么?如果 attention kernel 假设 K 矩阵是 `[seq, head_dim]` 连续布局(每行是一个 token 的 K\_h),那选项 Y 就是天然契合的——cache 里某个 head 的视图本身就是 `[seq, head_dim]` 连续。

### 2\. 第二个设计决策:状态机粒度——cache 是"全局一份"还是"每序列一份"

这个决策决定了 host 侧 KV cache 的生命周期和接口形态。

#### 选项 P:全局单序列(simple,推荐)

```c
// 整个进程只跟踪一条序列的状态
typedef struct {
    void *k_cache_dev;      // VRAM 地址,layer * kv_heads * max_seq * head_dim * 4
    void *v_cache_dev;      // 同上
    int cur_pos;            // 下一个要写入的 position(0 = 空 cache)
    int max_seq;            // 容量上限
    int num_layers;
    int num_kv_heads;
    int head_dim;
} kv_cache_t;

// 接口
kv_cache_t *kv_cache_create(int num_layers, int num_kv_heads, int head_dim, int max_seq);
void kv_cache_append(kv_cache_t *c, int layer, void *new_k, void *new_v);  // 写入 cur_pos 行,layer=N 时 cur_pos++
void kv_cache_reset(kv_cache_t *c);  // 重新开始一条序列
void kv_cache_destroy(kv_cache_t *c);
```

**特点**:

- 一次只处理一条序列,生成完一条 → reset → 生成下一条
- 不需要 batch 管理
- 接口最少

#### 选项 Q:多序列 / batch(generic)

```c
typedef struct {
    void *k_cache_dev;  // batch_size * layer * ... 
    int *cur_pos;       // 每条序列独立的 pos
    int batch_size;
    // ...
} kv_cache_batch_t;
```

**特点**:

- 支持并行生成多条序列
- 但我们当前 dispatch 是单 block 顺序执行,**根本没有 batch 并行性**
- 只会增加 host 侧复杂度,**不带来任何性能收益**

---

#### 我的判断

**强烈推荐选项 P** 。理由:

1. **YAGNI**:我们的 dispatch 是 `grid={1,1,1}` 顺序执行,batch 维度没有任何硬件并行性可利用——选项 Q 的复杂度是纯负担
2. **接口最小化原则(V11+ 沿用)**:派生量在内部算,接口字段越少越好
3. **可演进性**:未来如果要支持 batch,选项 P 加一层 `kv_cache_t *caches[N]` 数组就升级了,不会有架构债务
4. **toy 端到端目标对齐**:第三关 toy 模型只跑单序列推理,选项 Q 的能力一行都用不上

唯一 **不选 P 的理由**:如果你想顺便练习 batch 调度作为学习目标。但这违背"一次只引入一个新维度"的项目原则——KV cache 本身就是新维度,叠 batch 会让 debug 难度爆炸。

### 3\. 第三个设计决策:attention 读取路径——kernel 改还是 host 限制 dispatch 范围

这是 KV cache 项目的核心难点。先把问题讲清楚:

#### 没有 KV cache 时,attention 是怎么跑的

回顾我们现有的 `qkt_scale.S` 和 `pv.S` (基于 V19 文档,我没具体读过源码,先按推断走,后面要核对)。

attention 的核心数学:

```
scores = softmax(Q · K^T / sqrt(d))      # shape: [q_heads, seq_len]
output = scores · V                       # shape: [q_heads, head_dim]
```

之前的测试场景里,Q/K/V 都是 **同一次前向产生的完整矩阵**,seq\_len 是固定常量,kernel 直接读完整矩阵。

#### 有 KV cache 时,attention 要怎么变

decode 阶段(生成第 t 个 token):

- **Q 只有当前 token 的**:`Q_t` shape = `[q_heads, head_dim]` (注意:**只有 1 行**,不是 seq\_len 行)
- **K, V 是 cache 里 0..t 的全部**:`K_full` shape = `[kv_heads, t+1, head_dim]`,但 cache buffer 是 `[kv_heads, max_seq, head_dim]`
- 计算:

```
scores = softmax(Q_t · K_full^T / sqrt(d))    # [q_heads, t+1]
output = scores · V_full                       # [q_heads, head_dim]
```

**两个变化**:

1. Q 从矩阵退化成单行(从二维 reduce 到一维)
2. K/V 的有效长度是 `cur_pos+1`,但物理 buffer 是 `max_seq` —— **物理 buffer 后面那段是脏的、零的、或者上一条序列残留**

#### 三种实现路径

##### 路径 A:不改 kernel,host 侧"假装 cache 长度就是 cur\_pos+1"

每次 decode 前:

- 把 `K_cache[:, 0:cur_pos+1, :]` **拷贝** 到一块"刚好这么大"的 tight buffer
- 把这个 tight buffer 喂给现有 attention kernel

**优点**:零 kernel 改动,project 速度最快  
**缺点**:**每次 decode 多一次 D2D 拷贝** (且尺寸随 `cur_pos` 增长)。Qwen2-0.5B kv\_heads=2、head\_dim=64、24 layer,cur\_pos=2048 时拷贝约 24×2×2048×64×4×2 ≈ **96 MB** —— 比一次 GEMM 还重,**性能上不可接受**

##### 路径 B:不改 kernel,kernel 通过 args.seq\_len = cur\_pos+1 控制有效长度

如果现有 attention kernel 的 args 里有 `seq_len` 字段,而且 kernel 是"看 args.seq\_len,只算这么多行"——那么:

- host 侧不拷贝,直接把 `K_cache 整块的起始地址 + seq_len = cur_pos+1` 传给 kernel
- 但是 **K\_cache 物理 stride 是 `max_seq * head_dim`,有效长度是 `cur_pos+1`** —— 这俩不等
- 现有 kernel 可能假设 `K` 是 tightly packed 的(stride = seq\_len \* head\_dim),那就不能直接用,要改 kernel 让它读 `max_seq` 步长

**优点**:零拷贝,性能最好  
**缺点**:需要 kernel 支持"逻辑长度 ≠ 物理 stride"——可能要改 `qkt_scale.S` 和 `pv.S`,引入 `stride` 参数

##### 路径 C:第一次先拼一个 decode-only 的简化新 kernel

写一个新 kernel `attn_decode.S`,专门处理 "Q 是单 token + K/V 是 cache prefix" 的情况:

- Q 只有 `q_heads * head_dim` 这么大,直接全部 load 到寄存器/共享内存
- K/V 是 cache buffer,逻辑长度 `cur_pos+1`,物理 stride `max_seq*head_dim`
- 内联做 softmax(因为只有一个 query position,reduce 只跨 K 维)

**优点**:针对 decode 的最优结构,kernel 简单(没有 Q 的 batch 维)  
**缺点**:多写一个 kernel;之前的 `qkt_scale + softmax16 + pv` 三件套退化成 prefill 专用

#### 路径 C(写 decode 专用新 kernel):需要做什么

- **scores 只有 1 行** `[cur_pos+1]`,reduce 维度是 cur\_pos+1
- **output 只有 1 行** `[head_dim]`

新 kernel 结构:

- 32 lane 全部沿 K 维(seq 维)切。每 lane 处理 cur\_pos+1 中的一段
- 一次 dispatch 处理一个 head(用 grid.x = num\_heads 多 block 展开,或者 head 循环)
- 内部串起 qkt + scale + softmax + pv 的所有数学(可以是单 kernel,也可以是 2~3 个小 kernel)

**新 kernel 比通用化老 kernel 简单** ——因为 Q 单行让"行循环"维度消失了。

#### 路径 A(host 拷贝)修正

**路径 A 仍然事实上不可行**,这个判断不变。

### 设计层次 0:数学梳理(我们到底要算什么)

decode 第 t 步,输入和输出:

**已知**:

- `Q` shape = `[q_heads, head_dim]` —— 当前 token 的 Q,**只有一行 token**,但有 14 个 head
- `K_cache` shape = `[kv_heads, max_seq, head_dim]` —— 物理 buffer,只有前 `cur_pos+1` 行有效
- `V_cache` shape = `[kv_heads, max_seq, head_dim]` —— 同上
- `cur_pos` —— host 知道,作为 args 传入
- `scale = 1/sqrt(head_dim)`

**要算**:

- `O` shape = `[q_heads, head_dim]` —— 当前 token 的 attention 输出,也是单行 token、14 head

**GQA 映射**:14 query\_heads 共享 2 kv\_heads,每个 kv\_head 被 7 个 query\_head 共享。具体哪个 query\_head 用哪个 kv\_head:`kv_head_id = query_head_id / 7` (整数除)。

**对每个 query\_head `qh`** (独立计算):

```
kh = qh / 7                              # 选 kv_head
scores[j] = sum_k(Q[qh, k] * K_cache[kh, j, k]) * scale  for j in 0..cur_pos
P[j] = softmax(scores)                   # 沿 j 维 softmax
O[qh, k] = sum_j(P[j] * V_cache[kh, j, k])  for k in 0..head_dim
```

这就是 decode attention 的全部数学。 **注意三个特点**:

1. **每个 query\_head 之间完全独立** ——14 个 head 可以并行(顺序也行)
2. **q\_heads 维度上没有 reduce** ——每个 head 自己算自己的
3. **reduce 都在 K/V 维(j 维),长度是动态的 cur\_pos+1**

### 设计层次 1:把这些数学拆成几个 kernel?

有三个候选拆法:

#### 拆法 α:1 个 fused kernel —— "decode attention all-in-one"

一个 kernel 算完 `Q → scores → softmax → P → O` 整套。每次 dispatch 处理一个 query\_head(grid.x = num\_q\_heads = 14)。

**优点**:

- 中间结果 scores、P 不写回 VRAM,留在 lane 寄存器里 —— **零中间内存**,零中间 dispatch 开销
- host 侧只需 1 次 dispatch(`grid={14,1,1}`)
- 数学上紧凑,decode 这种小尺寸最吃这种 fusion

**缺点**:

- kernel 复杂度上升:Q load + qkt reduce + softmax(max + sum 两个 reduce + exp)+ pv reduce 全在一个 kernel 里
- debug 难度上升(任何一段错都看不到中间值,需要专门的 debug kernel)
- scores 数组要存在哪?cur\_pos+1 最大可能上千,一个 lane 寄存器装不下 —— **需要中间 buffer 或者用 VRAM 临时存**

#### 拆法 β:3 个 kernel —— qkt\_decode + softmax\_decode + pv\_decode

跟现有 prefill 三件套结构对应,但全部支持动态 seq\_len 和 head 循环。

**优点**:

- 结构上和 prefill 三件套对称,debug 时可以单独验证每一步
- scores 的存放在 VRAM 上很自然(就是 kernel 1 的 output、kernel 2 的 input)
- 每个 kernel 都比 fused 简单很多

**缺点**:

- 3 次 dispatch / decode step / layer 共 24 layer × 3 = **72 次 dispatch / token**
- scores 中间 buffer 需要分配(动态长度,需要按 max\_seq 预留)
- 中间结果 D2D 传输 —— 不,其实没有 D2D 拷贝,只是 VRAM 上的 buffer 传地址

#### 拆法 γ:2 个 kernel —— qkt+softmax fused + pv

折中。前两步 fused 成一个 kernel(Q·K^T 得到 scores 后立即在 lane 寄存器里 softmax,写回 VRAM 时已经是 P),pv 单独。

**优点**:

- scores 不需要 VRAM buffer(留在 lane 寄存器里 softmax 完直接写 P)
- 比 α 简单一些(pv 单独),比 β 少一次 dispatch
- softmax reduce 和 qkt 共用 lane 分工,有协同空间

**缺点**:

- "前两步 fused 后第三步分开"——结构稍微不对称
- 一个 kernel 同时干 qkt + softmax,设计复杂度介于 α 和 β 之间

---

#### 我的判断

**推荐拆法 γ(qkt+softmax fused + pv 分开)** 。理由:

1. **scores 的"短而临时"特性**:scores 长度是 cur\_pos+1,典型 < 1000,在 SIMT 32 lane 上每 lane 持有 ~30 个 fp32(寄存器够用,或者临时栈)。 **让 scores 不出 lane** 是最自然的——既不浪费 VRAM buffer,也不浪费 D2H/H2D 带宽
2. **softmax 必须有 max-reduce 和 sum-reduce 两次跨 lane reduce**:这两个 reduce 必须在 scores 还在 lane 寄存器里时做完,**或者** scores 写到 VRAM 然后另一个 kernel 重读做 reduce —— 后者就是拆法 β。 **β 的 softmax kernel 要重新设计 lane 分工策略,γ 的 fused kernel 让 lane 分工和 qkt 共用一致**,β 多了一份心智负担
3. **pv 单独**:pv 的 reduce 是沿 j 维(scores 维度),lane 分工自然不同——是沿 head\_dim 切 lane(每 lane 算 head\_dim 中的一段),**和 qkt+softmax 阶段沿 j 维切 lane 不一样** 。强行 fuse 进 α 会导致 lane 分工冲突
4. **可演进性**:如果未来发现 fused 太复杂,γ 可以退化成 β(把 fused 的两半再拆开);如果发现 dispatch 开销大,γ 也可以进化到 α(把 pv 也合进来)。 **γ 是中间灵活点**
5. **cur\_pos+1 上限分析**:Qwen2-0.5B 实际 max\_seq=32768,但 toy 阶段我们 max\_seq 远小于此(toy 几十,真实推理可以受限于 VRAM 先做 max\_seq=512 或 1024)。 **每 lane 持有 cur\_pos+1 / 32 个 fp32**,即使 max\_seq=1024,每 lane 32 个 fp32 = 128 字节,完全装得下(32 个 fp 寄存器我们有 ft0-ft11 + fa0-fa7 + fs0-fs11 = 32 个)—— **理论可行,但实际要谨慎** (寄存器数量和函数其他用途要错开)

**最大不确定**:γ 的 fused kernel 里 scores 究竟是放寄存器还是栈,要展开层次 2 才能确定。如果 scores 必须放栈或临时 VRAM,γ 相比 β 的优势会缩小。

### 一个我必须先想透的问题:scores 到底放哪

刚才推 γ 的时候我说"scores 留在 lane 寄存器里 softmax 完直接写 P"。 **但 P 也要写到某处** —— pv kernel 要读 P。所以 P 必须落盘到 VRAM。

那么真实问题是:

- **scores(softmax 前)** 留在哪?
- **P(softmax 后)** 留在哪?
- 两者是否是同一块内存?

**有三种处理方式**:

#### 方式 1:scores 留 lane 寄存器,P 写 VRAM

- qkt 阶段:32 个 lane 协作算出 scores,**每 lane 持有 (cur\_pos+1)/32 个 fp32**
- softmax 阶段:在 lane 寄存器里做 max-reduce、sub、exp、sum-reduce、div —— scores 一直没出寄存器
- 写 P:softmax 完毕后,每 lane 把自己持有的那段 fp32 写到 VRAM 上的 P buffer

**关键约束**:每 lane 要持有的 fp32 数量受寄存器数量限制。 `(cur_pos+1)/32` 需要 ≤ 可用寄存器数。

#### 方式 2:scores 走 VRAM buffer,P 也走 VRAM buffer(但可以是同一 buffer 复用)

- qkt 阶段:每 lane 算完后写到 VRAM scores buffer
- softmax 阶段:从 VRAM 读回来做 reduce,写回 VRAM(可以原地)
- pv 阶段:从 VRAM 读 P

**关键约束**:cur\_pos+1 没有上限(最多 max\_seq)—— 可扩展性最好。但 fused 的意义就削弱了(softmax 要重新读 scores)。

#### 方式 3:混合(scores 大时退化到方式 2,小时用方式 1)

太复杂,先不考虑。

### 寄存器预算实算

我们的 fp 寄存器一共 32 个:`ft0-ft11` (12 个)、 `fs0-fs11` (12 个)、 `fa0-fa7` (8 个)。

但是,kernel 里 **不能全用满** ——有些寄存器要保留给:

- scale(1 个)
- 当前 max / 当前 sum / 当前正在算的中间值(3~4 个)
- VSHFL\_XOR 的临时寄存器(1~2 个)

**保守估计每 lane 可持有的 scores 数量上限 ≈ 24 个 fp32** (留 8 个给临时变量)。

那么 **方式 1 可承受的 cur\_pos+1 上限** = 24 × 32 = **768** 。

- **真实 Qwen2-0.5B 推理**:max\_seq 设多少?V19 文档第 3 节说 `KV cache(seq=2048)` ✗ —— **超过 768 了**

### 我的判断

**先用方式 1** 。理由:

1. **toy 端到端跑通是当前目标**,toy seq\_len 必然远 < 768
2. **方式 1 真的很优雅**:fused 的语义贯彻到底,scores 不出寄存器,1 次 dispatch 完成 qkt + softmax
3. **真实推理 max\_seq 受限到 512 也能跑通**,先把 INT8、量化、performance 都解决再回来扩 max\_seq —— 这是合理的工程优先级
4. **方式 2 留作 fallback**:cur\_pos+1 > 768 时切回方式 2(本质就是退化成拆法 β)

**记一笔工程债务**:`max_seq > 768 时需要切换到 scores 走 VRAM 的版本,或者改用更激进的寄存器使用策略` 。

### 但这里有一个隐藏风险我必须暴露

**RV32 没有"动态索引访问寄存器"的指令** 。也就是说,如果 scores 存在 `ft0..ft23` 这 24 个寄存器里,你不能写 `flw ft[i], 0(addr)` 这种动态 i 索引——i 必须是编译时常量。

那么 qkt 算 scores 时,每 lane 沿着自己负责的那段 j 跑循环,**循环里每次得到一个新的 fp 值,这个值要写到哪个寄存器**?

**几种可能的处理方式**:

- **(a)** 循环展开:每 lane 持有的 24 个 scores 用 24 条独立指令算,每条指令显式写到 `ft0` 、 `ft1` 、…、 `ft23` 。 **但这样代码膨胀,而且 24 是上限,不同 cur\_pos 还得不同版本**
- **(b)** 用 VRAM 上的"per-lane scratchpad"做"软寄存器":每 lane 在 VRAM 上有一小块自己的临时区,fp 值 fsw 进去再 flw 出来—— **这等于方式 2 了,fused 优势消失**
- **(c)** 用栈(每 warp 一份):同样需要 fsw/flw,本质和 (b) 一样

**这个问题让方式 1 的"零中间内存"幻觉破产** 。RV32 没有 array-of-registers 的能力,**真的把 24 个 scores 留在寄存器里需要循环展开,而循环展开需要编译时已知 cur\_pos** 。

但 cur\_pos 是运行时变化的——每 decode 一步 +1。 **循环展开方案不可行** 。

### 重新评估

那么剩下的现实方案:

- **方式 1 实际执行需要 lane-local scratchpad** (VRAM 或栈),**等于方式 2 退化**
- **方式 2 直接用 scores VRAM buffer**

**两种方式的真实差别其实很小** —— 方式 1 的 scratchpad 也是 VRAM,方式 2 的 scores buffer 也是 VRAM。 **唯一区别是**:方式 1 是"per-lane 私有 VRAM 区"vs 方式 2 是"共享 scores buffer"。

而且方式 2 还有一个意外优点:**softmax 阶段可以是独立 kernel,debug 容易** —— 退化回拆法 β。

### 设计层次 2:lane 分工和中间 buffer 布局

现在我们有 3 个 kernel,每个独立设计 lane 分工。我先把 **3 个 kernel 各自要做什么 + buffer 接力关系** 摆清楚,然后再展开每个的 lane 分工。

#### 3 个 kernel 的接力图

```
输入                      输出
              -------                   -------
qkt_decode:   Q, K_cache, cur_pos  →   scores buffer [num_q_heads, cur_pos+1]
softmax_decode: scores buffer       →   P buffer       [num_q_heads, cur_pos+1]
pv_decode:    P, V_cache, cur_pos   →   O              [num_q_heads, head_dim]
```

**buffer 的物理大小怎么定?**

- scores 和 P 的逻辑形状是 `[num_q_heads, cur_pos+1]`,但 cur\_pos 每步都涨
- **物理 buffer 必须按 max\_seq 预留**:`[num_q_heads, max_seq]`
- 每 decode step 实际只用前 `cur_pos+1` 列,后面的列是脏数据,**kernel 必须用 cur\_pos 控制不读越界**

这在 host 侧体现为:**scores buffer 和 P buffer 是 KV cache 的"姐妹结构"**,生命周期与 KV cache 一致(整个序列共享一个 buffer,不是每 decode step 重新分配)。

---

#### 现在看每个 kernel 的 lane 分工

我会逐个展开。先看最直观的 **qkt\_decode**:

##### qkt\_decode 的数学

```
对每个 query_head qh ∈ [0, 14):
  kh = qh / 7
  对每个 j ∈ [0, cur_pos+1):
    scores[qh, j] = sum_k(Q[qh, k] * K_cache[kh, j, k]) * scale     # k ∈ [0, head_dim)
```

**外层是 (qh, j),内层是 k 维 reduce(长度 head\_dim=64)** 。

##### lane 分工候选

我们有 32 lane,要把 (qh, j) 这两个独立维度铺开。head\_dim=64 是 reduce 维度,**不是并行维度** (reduce 维度上每 lane 算 64/32=2 个元素然后 reduce)。

**候选 a:grid 展开 qh,32 lane 分 j** (我推荐)

- `grid={num_q_heads=14, 1, 1}`,每 block 处理一个 query\_head
- block 内 32 lane,每 lane 负责 j 维上的一段 `(cur_pos+1) / 32` 个 j
- 每 lane 内部:对自己负责的每个 j,做 head\_dim=64 的 dot product

**优点**:

- grid 自然切 head 维,无需 head 循环
- 每 lane 沿 j 维分段非常自然(类似 vmul 模板)
- cur\_pos+1 不是 32 倍数时,后面 lane 多算几个 j(tail 处理简单)

**缺点**:

- head\_dim=64 的 dot product 是 **单 lane 内串行** (64 次 fmadd)。 **没有跨 lane reduce** 。这是性能上的浪费——head\_dim 完全可以多 lane 协作

**候选 b:grid 展开 qh,32 lane 协作算 head\_dim 的 dot product**

- `grid={num_q_heads × ?, 1, 1}` 或者 head 循环
- 32 lane 沿 head\_dim 切(每 lane 算 64/32=2 个乘),做 reduce 得到一个 dot product 值
- 但是这样 **每次 dispatch 只算一个 (qh, j)**,需要 14 × (cur\_pos+1) 次 dispatch ≫ 14 次,完全不可接受

**候选 c:32 lane 同时切 j 和 head\_dim**

- 比如 4 lane 一组,每组算一个 j 的 head\_dim reduce;32 lane = 8 组同时算 8 个 j
- 复杂,而且 cur\_pos+1 不一定是 8 倍数

#### 我的判断:候选 a

理由:

1. **head\_dim 串行(64 次 fmadd)在数据规模上完全可以接受**:64 次 fmadd 是非常便宜的,而 cur\_pos 增长才是真负担(reduce 维度长度可能是 head\_dim 的几十倍)
2. **结构对应"vmul 单 warp + stride 内部循环"模板** (V18 验证过的模板),Eclipse 已经熟悉这套
3. **cur\_pos+1 < 32 时** 也好处理:lane >= cur\_pos+1 走"虚 lane",和 softmax16 的处理方式同源

##### 关键细节:cur\_pos+1 怎么分给 32 lane

设 `S = cur_pos + 1` (scores 长度)。需要把 S 分给 32 lane。

**方案 1**:`elems_per_lane = ceil(S/32)`,lane k 算 `j ∈ [k*elems_per_lane, min((k+1)*elems_per_lane, S))`

- 缺点:有些 lane 算 0 个 j(当 S < 32 时,后面 lane 全闲)

**方案 2** (stride=32 模板,vmul 用的):`for (j = lane_id; j < S; j += 32)`

- 优点:负载均衡好,代码模板熟悉
- 缺点:scores 写出去的位置是 `j = lane_id, lane_id+32, lane_id+64, ...`,**写出去的位置不连续** ——但写出去本来就是 random access,scatter 写没关系

**方案 2 完胜** 。这是 vmul 已经验证过的标准模板。

##### tail 处理:cur\_pos+1 不是 32 倍数时

stride=32 模板天然处理 tail —— `for (j = lane_id; j < S; j += 32)` 自动跳过越界 j。 **没有 tail 问题** 。

##### S < 32 时(序列开头)

cur\_pos=0(刚生成第 1 个 token)时,S=1,只有 lane 0 干活,其他 lane 一开始 j=lane\_id ≥ 1 > S 直接退出循环。 **自然处理** 。

### softmax\_decode 的设计

#### 数学

```
对每个 query_head qh ∈ [0, num_q_heads):
  m = max(scores[qh, 0..S-1])              # 跨 j 求 max
  for j in 0..S:
    e[j] = exp(scores[qh, j] - m)
  sum = sum(e[0..S-1])                     # 跨 j 求 sum
  for j in 0..S:
    P[qh, j] = e[j] / sum
```

**注意**:

- 每个 query\_head 完全独立(和 qkt\_decode 一样)
- reduce 维度是 j(长度 S = cur\_pos+1)
- **两次 reduce**:max 和 sum
- exp 用我们已有的 `vexp.s` 自定义指令

#### lane 分工

沿用 qkt\_decode 的策略:

- `grid={num_q_heads, 1, 1}`,每 block 处理一个 query\_head
- block 内 32 lane,每 lane 用 stride=32 模板分担 j
- **每 lane 持有自己负责的 S/32 个 scores**

但这里有一个 **新挑战**:每 lane 持有的 scores 数量 ≥ 1 个(j ∈ {lane\_id, lane\_id+32,...})。求 max 和 sum 时:

1. **lane 内**:先在自己持有的元素之间求 local\_max / local\_sum
2. **跨 lane**:用 VSHFL\_XOR 5 步树形 reduce 得到 global\_max / global\_sum

这个结构和 **rmsnorm.S** 的"local sum-of-squares + 5 步 XOR shuffle reduce" 是同构的。Eclipse 已经熟悉这套(milestone 3.1)。

#### 关键差异:scores 要读两次

数学上需要两次扫 scores:

1. 第一遍:求 max
2. 第二遍:用 max 算 e\[j\],累加 sum
3. 第三遍:用 sum 算 P\[j\] = e\[j\] / sum

**问题**:e 是中间值,如果不存下来,第三遍时还得重新算 exp(很贵)。

**两种处理**:

**方式 X:e 落 VRAM 暂存(原地写到 P buffer)**

- 第一遍:扫 scores 求 max
- 第二遍:扫 scores 算 exp(scores\[j\] - max),把 e\[j\] **写到 P buffer 对应位置** (原地暂存)
- 跨 lane reduce 求 sum(此时每 lane 持有自己的 local\_sum)
- 第三遍:从 P buffer **读回** e\[j\],除以 sum,**写回** P buffer 同一位置(原地)

**方式 Y:e 留寄存器(限制 S/32 ≤ 寄存器数)**

- 同 fused γ 那个方案,**RV32 索引限制让这条路死了**

**方式 X 是唯一可行** 。注意:**P buffer 在 softmax 内部充当暂存区**,这是项目里第一次出现"输出 buffer 兼临时区"——值得记一笔。

#### softmax\_decode 的 args

```c
struct softmax_decode_args {
    uint32_t scores_off;     // 输入 [num_q_heads, max_seq]
    uint32_t P_off;          // 输出 [num_q_heads, max_seq],也是中间暂存
    uint32_t cur_pos;        // S = cur_pos + 1
    uint32_t max_seq;        // 物理 stride
};
```

**4 个字段** 。比 qkt\_decode 少——因为 softmax 不涉及 GQA(每行独立处理)、不涉及 head\_dim、不涉及 scale。

#### 一个 corner case:S=1 时(decode 第一个 token,prefill 还没跑过)

S=1 时,只有 lane 0 持有一个元素,其他 lane 不持有任何元素。

**问题**:VSHFL\_XOR reduce 时,lane >=1 的 local\_max 该是什么?

- 求 max 用 `-INF` (中性元)
- 求 sum 用 `0` (中性元)

这和 softmax16.S 的"虚 lane 处理"思路完全一样。 **实现细节**:每 lane 在循环开始时初始化 `local_max = -INF, local_sum = 0`,如果循环一次都没跑(j 起点就 ≥ S),那 local\_max 就保持 -INF —— 自然作为 reduce 的中性元。

但这里有个 **真正的小坑**:S=1 时只有 lane 0 写一个元素,但 lane 0 也会被卷入 reduce —— 它会和 31 个 -INF / 0 做 reduce,**结果还是 lane 0 自己的值** 。✓ 数学正确。

第二遍写 e\[j\] 也只有 lane 0 写一次。✓ 第三遍读 e\[j\] / sum 也只有 lane 0 做。✓

**S=1 不需要特殊处理**,自然落到主路径。

### pv\_decode 的设计

#### 数学

```
对每个 query_head qh ∈ [0, num_q_heads):
  kh = qh / q_per_kv
  for k in 0..head_dim:
    O[qh, k] = sum_j(P[qh, j] * V_cache[kh, j, k])    # j ∈ [0, S)
```

**注意**:

- 每个 query\_head 独立(同前)
- 外层 (qh, k),reduce 维度是 j(长度 S)
- O shape = `[num_q_heads, head_dim]` —— 每个 head 输出 head\_dim 个值

#### lane 分工

这里 **结构变了**:reduce 维度是 j(长度 S),但 **真正铺开的并行维度是 k(head\_dim)** ——每个 (qh, k) 独立,共 `num_q_heads * head_dim = 14 * 64 = 896` 个独立 dot product。

**候选**:

- **(i)** `grid={num_q_heads, 1, 1}`,32 lane 沿 k 切(每 lane 算 head\_dim/32 = 2 个 k):每 lane 内部对自己的 2 个 k 做 j ∈ \[0,S) 的 reduce(串行)
- **(ii)** `grid={num_q_heads, 1, 1}`,32 lane 沿 j 切(stride=32),需要跨 lane reduce 得到每个 k 的最终值

##### 候选 (i):32 lane 沿 k 切

每 lane 算 `head_dim / 32 = 2` 个 k(head\_dim=64 的情况下)。每个 k 内部:

```
对 lane k_local ∈ [0, 32):
  对 k_local 持有的每个 k(本地 2 个):
    acc = 0
    for j in 0..S:
      acc += P[qh, j] * V_cache[kh, j, k]
    O[qh, k] = acc
```

**没有跨 lane reduce**!每 lane 各算各的。

**优点**:

- 结构最简单(没有 reduce)
- 写 O 时位置不冲突(lane 0 写 k=0,1,lane 1 写 k=2,3,...)
- 与 head\_dim=64 完美对齐(32 × 2 = 64)

**缺点**:

- head\_dim 不是 32 倍数时(比如 head\_dim=63)就要 tail handling —— **Qwen2 head\_dim=64 没问题,但记下债务**
- 每 lane 内部对 j 串行 reduce 是 O(S) —— 但 S 长度是必须扫的,这是数学本质

##### 候选 (ii):32 lane 沿 j 切

每 lane 算 j 维上的一段,对每个 k 做 partial sum,跨 lane reduce 得到总和。

**问题**:每个 (qh, k) 都要做一次跨 lane reduce —— 64 次 reduce / dispatch,**比 (i) 慢得多** 。否决。

#### 我的判断:候选 (i)

**最简单结构,head\_dim=64 完美对齐 32 lane,没有跨 lane reduce** 。debug 友好。

### 暂停 + 几个想和你对齐的点

**1\. 一致性检查:scores 和 P 是否要分 buffer?**

我们目前的设计是 `scores` 和 `P` 是 VRAM 上 **两块独立的 buffer** (qkt 写 scores,softmax 读 scores 写 P,pv 读 P)。

但是,**softmax\_decode 内部用 P buffer 暂存 e\[j\]** ——这意味着 P buffer 在 softmax 执行过程中被覆盖。

这有个隐患:**如果 softmax 第二遍写 e\[j\] 时,某个 lane 还在第一遍读 scores —— 不会发生,因为他们读的是 scores buffer 不是 P buffer** 。✓ 没问题。

但 **还有一个简化**:softmax 既然不会读到 scores 第二次以后(第二遍是写 e\[j\] 到 P buffer 后才求 sum,sum 求完就只读 P buffer 不再读 scores 了),那 **scores 和 P 完全可以是同一块 buffer** (softmax in-place)?

让我推一下:

- 第一遍:每 lane 读 scores\[j\],累计 local\_max。 **只读 scores**
- 跨 lane reduce 得 global\_max
- 第二遍:每 lane 读 scores\[j\],算 e = exp(scores\[j\] - global\_max),写到 **同一位置** 。 **读旧 scores 写新 e**
- 跨 lane reduce 得 global\_sum(每 lane 的 local\_sum 是第二遍累加的)
- 第三遍:每 lane 读 e\[j\],算 P\[j\] = e\[j\] / global\_sum,写到同一位置

**第二遍读写同一位置**:每 lane 读自己负责的 j 的 scores,写同一个 j 的 e —— **不冲突** (同一 lane 的同一 j,串行读写)。 **第三遍同理** 。

**结论**:scores 和 P 可以是同一块 buffer,softmax in-place。

这把"scores buffer + P buffer"两块合并成一块——节省 max\_seq × num\_q\_heads × 4 字节。Qwen2 max\_seq=512, q\_heads=14 时是 28KB,小数,但 **接口更干净** 。

**对应的设计变更**:

- pv 的 args 里 `P_off` 改名为 `scores_off` (同一块 buffer)
- softmax 的 args 里 `scores_off` 和 `P_off` 合并为一个字段

**重新统计** softmax\_decode args:

```c
struct softmax_decode_args {
    uint32_t scores_off;     // in-place,既是输入也是输出
    uint32_t cur_pos;
    uint32_t max_seq;
};
```

**3 个字段** 。

**2\. 一个我想和你确认的设计原则**

我刚才说"scores 和 P 同 buffer"算是一种 in-place 简化,**但 V19 文档总结过 in-place 三层安全性论证** (单 lane 内寄存器暂存 / 跨 lane 不重叠 / 跨循环维度不重叠)。

应用到这里:

- **单 lane 内**:第二遍读 scores\[j\] 后,**算完** e 才写 同一 j —— 寄存器暂存了 ✓
- **跨 lane**:每 lane 负责的 j 不重叠 ✓
- **跨循环维度**:这里没有"循环维度"重叠的问题(softmax 三遍都是一维 j 维度)✓

三层安全性都成立。 **in-place 安全** 。

#### 状态机的核心:cur\_pos 在哪儿、谁更新它

回顾我们之前的决策:**全局单序列(选项 P)** 。所以全局只有一个 `cur_pos`,但是这个 `cur_pos` 该住在哪?

**候选 1:住在 host 内存** (C 结构体字段)

```c
typedef struct {
    void *k_cache_dev;
    void *v_cache_dev;
    void *scores_p_buffer_dev;
    int cur_pos;          // host 内存
    int max_seq;
    int num_layers;
    int num_kv_heads;
    int num_q_heads;
    int head_dim;
} kv_cache_t;
```

每次 dispatch 把 `cur_pos` 拷进 args buffer 一起传给 device。

**候选 2:住在 device 内存** (VRAM 上一个 4 字节字段)

---

#### 增量写入语义:K\_t / V\_t 怎么进 cache

decode 第 t 步流程:

1. host 准备输入 token(embedding lookup → rmsnorm →...)
2. 每一层:
	- Q/K/V projection → 得到 `Q_t`, `K_t`, `V_t` (都是 `[num_q_heads, head_dim]` 或 `[num_kv_heads, head_dim]`)
		- **K\_t / V\_t 写入 cache 的第 cur\_pos 行**
		- RoPE(对 Q\_t 和 K\_t,但 K\_t 是已经在 cache 里还是即将写入?——这是关键问题)
		- attention(读 cache 0..cur\_pos)
		- O proj、residual、MLP、...
3. 所有层算完后 → lm\_head → argmax → 得到 token t+1
4. **cur\_pos += 1**

**第 2 步里 RoPE 的位置是关键问题** 。两种顺序:

**顺序 A:RoPE 后写 cache**

```
Q_t = Q_proj(...)
K_t = K_proj(...)
V_t = V_proj(...)
RoPE(Q_t, K_t, token_pos=cur_pos)        # 都是新算的,在 host 知道的 buffer 上
write_cache(K_t, V_t, position=cur_pos)  # 把已经 RoPE 过的 K_t 写进去
attention(Q_t, K_cache, V_cache)          # K_cache 里已经包含了 RoPE 过的 K
```

**顺序 B:写 cache 后 RoPE**

```
Q_t = Q_proj(...)
K_t = K_proj(...)
V_t = V_proj(...)
write_cache(K_t, V_t, position=cur_pos)  # 写没 RoPE 过的 K
RoPE(Q_t, ???)                            # K 已经在 cache 里了,要对哪个 K 做 RoPE?
```

**顺序 A 是正确的**,顺序 B 有歧义(RoPE 只对当前 token 的 Q/K 做,但 cache 里历史的 K 已经 RoPE 过了,不能重复 RoPE)。

**结论**:

- **K\_t 写入 cache 之前必须已经 RoPE 过**
- **写入 cache 的是 RoPE 后的 K\_t**
- 历史的 K\_0..K\_{t-1} 在它们各自的时间点已经各自 RoPE 过了(用各自的 token\_pos)
- attention 读 cache 时,读到的就是已经 RoPE 过的所有 K —— ✓ 正确

这意味着 KV cache 接口需要的核心操作就是:

**`kv_cache_write(layer, position, K_t, V_t)`** —— 把已处理好的 K\_t/V\_t 写入 cache 第 `position` 行

这让"实现 X 还是 Y"的取舍稍微复杂了一点。

### 进入设计层次 3:host 侧 KV cache 接口

现在我们有足够信息把 KV cache 的 host 侧接口完整画出来了。

#### 数据结构

```c
typedef struct {
    /* device 上的 buffer */
    uint64_t k_cache_off;        // num_layers * num_kv_heads * max_seq * head_dim * 4 bytes
    uint64_t v_cache_off;        // 同上
    uint64_t scores_p_buffer_off; // num_q_heads * max_seq * 4 bytes (per-step,不分 layer)
    uint64_t o_buffer_off;        // num_q_heads * head_dim * 4 bytes (per-step)

    /* 形状参数 */
    int num_layers;
    int num_kv_heads;
    int num_q_heads;
    int head_dim;
    int max_seq;

    /* 状态 */
    int cur_pos;                  // 下一个要写入的 position(初始 0)
} kv_cache_t;
```

#### 接口函数

```c
/* 创建:分配所有 device buffer,初始化 cur_pos=0 */
int kv_cache_create(gpgpu_ctx *ctx, kv_cache_t *kv,
                    int num_layers, int num_kv_heads, int num_q_heads,
                    int head_dim, int max_seq);

/* 写入第 layer 层的 K/V 到当前 cur_pos 行 */
/* k_t_off: device 上一块 [num_kv_heads, head_dim] 的 buffer(已 RoPE 过) */
/* v_t_off: device 上一块 [num_kv_heads, head_dim] 的 buffer */
int kv_cache_write_layer(gpgpu_ctx *ctx, kv_cache_t *kv, int layer,
                         uint64_t k_t_off, uint64_t v_t_off);

/* 全部层写完后,推进位置 */
void kv_cache_advance(kv_cache_t *kv);

/* 重置:cur_pos = 0,buffer 内容不清(下次写入会覆盖) */
void kv_cache_reset(kv_cache_t *kv);

/* 销毁 */
void kv_cache_destroy(gpgpu_ctx *ctx, kv_cache_t *kv);
```

#### 关键设计选择我想和你对齐

##### 选择 (1):每层独立 write\_layer vs 一次性 write\_all\_layers

我选了 **每层独立** 。理由:

- transformer 是层串行的:layer 0 算完才能算 layer 1。 **K\_t / V\_t 是每层各自的**,不是一次产出 24 层
- "一次性写所有层" 没有意义,因为算下一层时上层还没产出
- 每层 write 后立即被本层 attention 读到,是自然的执行顺序

##### 选择 (2):cur\_pos 推进时机

`kv_cache_advance` 单独是一个函数,**在所有层都写完之后调一次** 。

为什么不在 `write_layer` 里 `cur_pos++`?**因为 24 层用的是同一个 `cur_pos`** ——layer 0 写第 cur\_pos 行,layer 1 也写第 cur\_pos 行(只是不同 layer 的 K cache),最后所有层都写完,cur\_pos 才推进到 cur\_pos+1。

##### 选择 (3):scores\_p\_buffer 和 o\_buffer 是否纳入 kv\_cache\_t

这两个 buffer **不是 cache 本身** (每 step 重写),但生命周期和 KV cache 一致(整个序列共享)。

两种 design:

- **A**:`kv_cache_t` 包揽所有"序列级 buffer"——KV cache + scores + o
- **B**:`kv_cache_t` 只管 KV,scores 和 o 单独管理

我倾向 **A** (包揽),理由:

- 这些 buffer 都是 **per-sequence 的** (reset 时一起清,destroy 时一起释放)
- 接口更简洁:host 主循环不需要单独管理 scores\_p\_buffer / o\_buffer 的生命周期
- "attention 中间 buffer" 概念上属于 KV cache 的配套设施

但 **B 也合理**:模块化更彻底,kv\_cache\_t 名实相符。

#### 主循环的样子(预览,让你看接口好不好用)

decode 单 step:

```c
/* 输入:当前 token id,kv_cache_t *kv */

/* 1. embedding lookup → x */
embedding_lookup(ctx, embed_table_off, token_id, x_off);

for (int layer = 0; layer < kv->num_layers; layer++) {
    /* 2. pre-attn rmsnorm */
    rmsnorm(ctx, x_off, ln1_w_off[layer], x_norm_off);

    /* 3. Q/K/V projections */
    gemm(ctx, x_norm_off, q_proj_w_off[layer], q_t_off);  /* [num_q_heads, head_dim] */
    gemm(ctx, x_norm_off, k_proj_w_off[layer], k_t_off);  /* [num_kv_heads, head_dim] */
    gemm(ctx, x_norm_off, v_proj_w_off[layer], v_t_off);  /* [num_kv_heads, head_dim] */

    /* 4. RoPE on Q, K (using kv->cur_pos as token_pos) */
    rope(ctx, q_t_off, sincos_off, kv->num_q_heads, kv->head_dim, kv->cur_pos);
    rope(ctx, k_t_off, sincos_off, kv->num_kv_heads, kv->head_dim, kv->cur_pos);

    /* 5. write K_t / V_t to cache */
    kv_cache_write_layer(ctx, kv, layer, k_t_off, v_t_off);

    /* 6. attention (3 kernel:qkt + softmax + pv) */
    qkt_decode(ctx, q_t_off, kv->k_cache_off + layer*K_LAYER_STRIDE,
               kv->scores_p_buffer_off, kv->cur_pos, ...);
    softmax_decode(ctx, kv->scores_p_buffer_off, kv->cur_pos, kv->max_seq);
    pv_decode(ctx, kv->scores_p_buffer_off, kv->v_cache_off + layer*V_LAYER_STRIDE,
              kv->o_buffer_off, kv->cur_pos, ...);

    /* 7. O proj + residual + MLP + ... */
    /* ... */
}

/* 8. advance position */
kv_cache_advance(kv);

/* 9. lm_head + argmax → next token */
```

这里面有一个 **重要的检查**:

- `kv->cur_pos` 在所有 24 层中保持不变
- 同一个 `cur_pos` 既被 RoPE 用作 token\_pos,也被 KV cache 用作写入行号,也被 attention 用作 cache 有效长度
- **所有层结束后才 advance**

接口够用,语义清晰。

### 一个我必须暴露的 milestone 范围漂移问题

V19 文档第 3 节明确说 KV cache milestone 是:

> **第二关剩余:KV cache 管理** (下次会话开工候选)
> 
> - 不是 kernel,是 host 侧 sequence position 状态机 + cache buffer 调度

**"不是 kernel"** 是 V19 给的明确定位。但我们的设计方案最后包含了 **3 个新 kernel** ——已经远超原 milestone 范围。

这是怎么发生的?**因为现有 attention 三件套是 prefill 写死的,KV cache 场景下不能直接用** ——这是我读 `qkt_scale.S` 那一关之后才暴露出来的事实。所以严格说,这不是"范围蔓延",是"原 milestone 设计时低估了依赖关系"。

但事到如今,我想和你确认怎么处理:

#### 选项 A:照原计划走完(推荐)

完成 KV cache + 3 decode kernel,作为 milestone 3.5 第二关剩余的扩展定义。 **实质上是"KV cache 管理 + 它需要的所有前置 kernel"作为一个 milestone** 。

- **优点**:有 KV cache 又有可用的 decode attention,真正构成了"decode 一个 token 的完整能力"
- **缺点**:工作量大,会话可能跨越多个 turn;3 个 kernel 同时调试 debug 难度叠加

#### 选项 B:拆成两个连续 milestone

- **milestone 3.5.2.a**:3 个 decode attention kernel(独立验证,host 侧手工构造 K\_cache、V\_cache、cur\_pos)
- **milestone 3.5.2.b**:KV cache 管理 + 集成 RoPE / 写 cache / 调用 decode attention

每个 milestone 单独完成 + handover。

- **优点**:每个 milestone 单独成立、单独可验证,debug 边界清晰(kernel 算法对错 vs cache 状态机对错可以分开定位)
- **缺点**:看起来"慢"了;handover 文档要写两份;但其实 **总工作量没变,只是切了块**

#### 我的判断:选 B

理由:

1. **debug 边界清晰**:kernel 数学有 bug vs cache 状态机有 bug,两类问题的诊断方法、定位工具完全不同。混在一起调试 = 噪音放大。这正是 V18 vmul 跳档教训的一般化—— **"一次只引入一个新维度"**
2. **decode kernel 能独立验证**:host 侧手工填充 K\_cache、V\_cache,各种 cur\_pos 值,纯 attention 数学验证。这一关跑通后,我们对 attention 数学的信心应该是"已知正确",再上 cache 状态机时 bug 范围被压缩到 host 侧
3. **handover 节奏**:如果 milestone B(cache 状态机)出现意外阻塞,A(decode kernel)已经收工沉淀,损失少
4. **符合项目"每 milestone 一个新维度"的节奏**:RoPE milestone 也是单一主题,clean handover 后续好维护

**唯一不选 B 的理由**:如果你觉得"反正都要跑端到端,一起做更整体感强"。但项目历史上,**V11~V19 每个 milestone 都是单一新主题**,这个节奏到现在保持得很稳定。
